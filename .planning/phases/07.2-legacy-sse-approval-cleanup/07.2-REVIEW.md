## Summary

Reviewed the Phase 07.2 backend, frontend, tests, active docs, and Excalidraw source for legacy SSE approval cleanup. The old FastAPI approval routes and desktop bridge/provider/dialog wiring are removed from active source, and the official AG-UI approval card path remains covered by targeted tests.

One blocking correctness issue remains: `create_paladin_agent()` does not fail fast on explicit `hitl.mode = "legacy_sse"` because it validates the mode only after model/workspace/MCP initialization. Two active documentation warnings also remain where docs still point future workers at deleted tests or UI behavior that Phase 07.2 intentionally removed.

## Critical Issues

### CR-01: Explicit legacy_sse can be masked by earlier agent initialization failures

**Classification:** BLOCKER
**File:** `apps/agent/src/agent/paladin_agent.py:329`
**Issue:** `create_paladin_agent()` reads the raw config at lines 314-318, but it does not call `_approval_mode(raw_config)` until line 358. Before that check, it loads model configs, resolves/validates the primary API key, creates workspace and skills directories, and loads MCP servers. This violates the Phase 07.2 fail-fast requirement for explicit `hitl.mode = "legacy_sse"`: a config that contains both `legacy_sse` and a missing API key will raise the API-key error first, and configured MCP setup may run before the unsupported legacy mode is rejected.
**Fix:**
```python
# After raw_config is parsed, before load_models(), _create_model(), workspace creation, or MCP loading:
mode = _approval_mode(raw_config)

# Later, reuse mode for logging instead of recomputing it.
```
Add a regression test around `create_paladin_agent()` using a config with `hitl.mode = "legacy_sse"` and an unset API key, asserting the raised `ValueError` names `legacy_sse` and `agui_interrupt`.

## Warnings

### WR-01: Active plan still tells workers to run a deleted backend test file

**Classification:** WARNING
**File:** `docs/superpowers/plans/2026-07-01-ag-ui-tool-approval.md:176`
**Issue:** The active plan command still runs `uv run pytest tests/test_agent.py tests/test_hitl.py -v`, but Phase 07.2 deletes `apps/agent/tests/test_hitl.py`. A worker following this plan will hit a collection error instead of the intended approval-mode verification.
**Fix:** Replace the command with the current active coverage, for example:
```bash
cd apps/agent
uv run pytest tests/test_agent.py tests/test_server.py tests/test_agui_dependency_probe.py -v
```

### WR-02: Active design doc still allows toolbar/modal approval UI that 07.2 forbids

**Classification:** WARNING
**File:** `docs/superpowers/specs/2026-07-01-ag-ui-tool-approval-design.md:143`
**Issue:** The design doc says the toolbar may keep a pending-approval indicator and that `ApprovalDialog` may remain useful if opened from the card. Phase 07.2's UI contract explicitly removes toolbar pending approval status and forbids rendering `ApprovalDialog` or adding a replacement official pending-status toolbar UI. Leaving this in active guidance risks reintroducing the UI path the phase just removed.
**Fix:** Reword the section to state that `AguiApprovalInterrupt`/`ApprovalCard` in the chat stream is the only active approval surface after 07.2, with no toolbar pending indicator and no `ApprovalDialog` ownership or fallback.
