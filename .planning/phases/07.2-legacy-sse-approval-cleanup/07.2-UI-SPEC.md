# Phase 07.2 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | shadcn |
| Preset | b2fA; style `base-nova`; base/theme `neutral`; radius `default`; menu accent `subtle` |
| Component library | shadcn/base components already installed: `button`, `alert-dialog`, `scroll-area`, `sheet`, `sonner`; do not add new components for this cleanup |
| Icon library | lucide |
| Font | Geist Variable via `@fontsource-variable/geist`; `font-sans` and `font-heading` inherit Geist |

Design-system source: `apps/desktop/components.json`, `apps/desktop/src/index.css`, and `cd apps/desktop && npx shadcn info`.

Phase-specific contract:
- Remove legacy `ApprovalBridge`, `ApprovalDialog`, `ApprovalProvider`, and `useApprovalContext` from active desktop source.
- Remove legacy approval status from `ChatToolbar` without replacing it with a new official pending-status toolbar UI.
- Preserve `AguiApprovalInterrupt` ownership of official approval interaction inside the chat area.
- Preserve `ApprovalCard` as the approval surface rendered by `AguiApprovalInterrupt`; do not redesign it in Phase 07.2.
- Keep `ChatToolbar` visually stable for non-approval context and tool-call information.

---

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, compact inline controls, toolbar micro-gaps |
| sm | 8px | Compact element spacing, card action gaps, list item spacing |
| md | 16px | Default component padding; `ApprovalCard` padding remains `p-4` |
| lg | 24px | Dialog/panel section padding when existing shadcn components require it |
| xl | 32px | Empty-state and centered layout padding |
| 2xl | 48px | Major vertical centering gaps only |
| 3xl | 64px | Page-level spacing only; not needed for this cleanup |

Exceptions: Existing shadcn/base component internals may retain their generated compact sizes, including `h-6`, `h-7`, `h-8`, `h-9`, `px-2.5`, `gap-1.5`, and icon sizes `size-3`, `size-3.5`, `size-4`. Do not introduce new spacing exceptions in Phase 07.2.

Phase-specific layout rules:
- Deleting legacy approval status must not change the `ChatToolbar` width (`w-56`), border treatment, or surrounding three-column layout.
- `ApprovalCard` spacing remains `my-2`, `p-4`, internal `gap-2`, metadata block `px-3 py-1.5`, and action buttons `h-8 px-3`.
- Removing `ApprovalProvider` from `App.tsx` must not add wrapper elements, extra margins, or layout shifts around `CopilotKitProvider`.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 | 1.5 |
| Label | 12px | 500 | 1.4 |
| Heading | 20px | 600 | 1.2 |
| Display | 24px | 600 | 1.2 |

Typography contract:
- Use existing Geist Variable font tokens only; no new font family.
- Limit this phase to two weights in new or touched UI code: regular `400` and semibold/medium `500-600` as already used by shadcn/base controls.
- Existing monospace identifiers in toolbar/tool metadata may continue to use `font-mono`.
- Approval-card labels remain compact: status label `text-xs font-medium`, tool name `text-sm font-semibold font-mono`, metadata/code `text-xs`.
- `ChatToolbar` labels remain `text-xs`; do not promote approval state into heading/display text.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `var(--background)` / `bg-background` | App shell, chat area, toolbar base surface |
| Secondary (30%) | `var(--muted)`, `var(--card)`, `var(--border)` | Panels, cards, metadata blocks, borders, subtle inactive states |
| Accent (10%) | `var(--primary)` plus existing approval semantic accents | Primary controls, selected states, existing approval-card approve/status affordances only |
| Destructive | `var(--destructive)` | Deny action and destructive-only affordances |

Accent reserved for: primary shadcn buttons, selected/current states, existing `ApprovalCard` approval affordance, and existing semantic success/warning badges inside approval/diff/status UI. Accent is not reserved for every interactive element.

Phase-specific color rules:
- Remove the legacy `ChatToolbar` pending approval block and its yellow status treatment with no replacement block.
- Preserve `ApprovalCard` status badge styling: `bg-amber-500/10`, `text-amber-700`, `dark:text-amber-400`.
- Preserve `ApprovalCard` approve styling: `bg-emerald-600`, hover `bg-emerald-700`, white text.
- Preserve `ApprovalCard` deny styling: `border-destructive/30`, `bg-destructive/10`, `text-destructive`, hover `bg-destructive/20`.
- Do not introduce a new approval palette, gradient, notification strip, toolbar badge, or top-level status color in this phase.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | `Approve` |
| Empty state heading | No approval-specific empty state is rendered |
| Empty state body | When there is no official AG-UI interrupt, render nothing; keep existing chat/toolbar empty states such as `暂无工具调用` unchanged |
| Error state | Existing CopilotKit/toast errors remain: `Agent 运行失败`, `Agent 内部错误`, `工具参数解析错误`, `工具执行失败`; do not add legacy approval error copy |
| Destructive confirmation | `Deny`: no confirmation dialog in Phase 07.2; click sends an official AG-UI denial payload `{ approved: false, reason: "User denied this tool call." }` |

Additional copywriting rules:
- Keep approval action labels in `ApprovalCard` as `Approve` and `Deny` for compatibility with existing tests and screenshots.
- Do not add toolbar copy such as `等待审批中...` for official pending interrupts in this phase.
- Remove legacy-only copy tied to request-id/EventSource approval status from active desktop UI.
- Keep `ChatToolbar` non-approval labels unchanged: `上下文`, `模型`, `消息数`, `线程`, `工具调用`, `暂无工具调用`, `清空对话`.

## Registry Safety

| Registry | Blocks Used | Safety Gate |
|----------|-------------|-------------|

Registry contract:

- If an executor discovers a new component is unavoidable, stop and route back through UI review before adding it.

## Interaction Contract

| Surface | Required Behavior |
|---------|-------------------|
| `AguiApprovalInterrupt` | Remains mounted in `ChatArea` and subscribes to official AG-UI run-finished interrupts. It renders only interrupts with `reason === "tool_call"`. |
| `ApprovalCard` | Remains the only approval UI. It displays tool name, optional argument summary, optional message/reason, and exactly two actions: `Deny` then `Approve`. |
| Approval decision | On first click, disable repeat sends via existing one-shot behavior, update local card status, and submit `agent.runAgent({ resume })` with a single `ResumeEntry`. |
| Denial payload | Must remain `{ approved: false, reason: "User denied this tool call." }`. |
| Approval payload | Must remain `{ approved: true }`. |
| `ChatToolbar` | Continues showing model/thread/message count/tool-call information and the clear conversation action; it must not read approval context or show approval pending status. |
| App shell | Removing `ApprovalProvider` must not change titlebar, sidebars, right panel, status bar, CopilotKit provider, or toast placement. |

Must-not interactions:
- Must not open `EventSource("/approval/stream")`.
- Must not `POST` to `/approval/{request_id}`.
- Must not render `ApprovalDialog`.
- Must not mount `ApprovalProvider`.
- Must not create a new official AG-UI pending-status toolbar UI.
- Must not move approval interaction outside the chat interrupt element.

## Accessibility Contract

| Requirement | Contract |
|-------------|----------|
| Keyboard | Existing `button` elements in `ApprovalCard` remain keyboard-focusable; disabled state must be applied after a decision is sent. |
| Focus | Removing `ApprovalDialog` removes modal focus trapping; do not add replacement focus traps for this phase. |
| Screen-reader semantics | Keep native buttons for `Approve` and `Deny`; preserve visible labels. |
| Contrast | Use shadcn token colors and existing semantic approval/destructive colors; do not introduce low-contrast custom colors. |
| Motion | No new animations or transitions beyond existing button color transitions. |
