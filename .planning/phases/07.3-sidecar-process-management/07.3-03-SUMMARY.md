## ConfigError variants 与 validate() 规则

```rust
pub enum ConfigError {
    NotFound(PathBuf),         // 配置文件不存在
    IoError(std::io::Error),   // 文件存在但读失败 (权限/磁盘)
    ParseError(serde_json::Error), // JSON 语法错或类型不匹配 (含 cmd 单串、port u16 溢出、缺必填字段)
    InvalidSchema(String),     // JSON 合法但字段值越界 (业务约束失败)
}
```

`validate()` 规则(失败返回 `Err(String)` 错误消息,由 `load_from_path` 包装为 `InvalidSchema`):

| 检查项 | 约束 | 失败消息示例 |
|---|---|---|
| `backoff_secs.len()` | ≤ 5 (SPEC 锁定) | `"backoff_secs length 6 > 5"` |
| `max_restarts` | ≤ 5 (SPEC 锁定) | `"max_restarts 99 > 5"` |
| `shutdown_grace_secs` | 1..=60 | `"shutdown_grace_secs 0 out of range (1..=60)"` |
| 每个 process 的 `cmd` | 非空 Vec | `"Agent cmd empty"` |
| 每个 process 的 `port` | != 0 (u16 已封顶 65535) | `"Agent port 0"` |
| `liveness.expect_status` | 100..=599 | `"Agent liveness expect_status 999 out of range (100..=599)"` |
| `readiness.expect_status`(若存在)| 100..=599 | `"Server readiness expect_status 99 out of range (100..=599)"` |

## Decisions Made

- **ConfigError 不引入 thiserror**:Cargo.toml 当前无 thiserror 依赖,引入只为节省 8 行手写 Display/Error 会增加一个传递依赖。手写 impl 与项目"最小依赖"原则一致(plan 01 D-13 不引入 tauri-plugin-shell 同款决策)。
- **`validate()` 返回 `Result<(), String>` 而非自定义 Error**:错误消息是自由文本(format!),无需 enum 结构化;`load_from_path` 把 String 包装为 `InvalidSchema(String)`。这让单元测试可直接断言消息内容,且 plan 06 supervisor 拿到的 InvalidSchema 即是人类可读诊断。
- **port=65536 在 u16 类型层就被拒绝(ParseError),不到 validate**:这与 plan 01 已锁定的 `pub port: u16` 字段类型一致 — u16 max=65535。validate 只需检查 `port == 0`(另一个常见错误)。这是双层防御:type 层拒绝溢出,validate 拒绝 0。
- **shell 元字符扫描不在 validate 范围**:命令注入面收敛靠 (1) `cmd` 字段 Vec<String> 类型层 + (2) supervisor 用 `tokio::process::Command::args(cmd)` 默认不走 shell。validate 只查 `cmd.is_empty()`,不查元字符 — 否则会误判合法参数(如 grep 模式中的 `;`)。held-out 测试 `test_cmd_with_shell_metacharacters_preserved_as_literal_arg` 显式证明此设计。
- **`#![allow(dead_code)]` module-level 抑制**:与 plan 02 state_machine.rs 同款。本模块的 `ConfigError`/`load_from_path`/`validate` 在 plan 06 supervisor 集成前仅被测试调用,集成后 warning 自然消失。
