**Gathered:** 2026-07-05
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 07.3 让 Tauri 桌面端成为 Python Agent 与 Go Server 两个子进程的唯一托管者：在 `setup()` 中 spawn 子进程、周期性健康探针、崩溃时按锁定退避序列自动重启、退出时优雅关闭、捕获 stdout/stderr 推前端日志面板并落盘、暴露手动控制命令、把前端 URL 从硬编码迁到 Rust 配置驱动。

本 phase 只做 **dev 模式 spawn**（`uv run paladin-agent serve` / `go run ./cmd/server`），不做 Python 二进制冻结、不配 Tauri `externalBin`——那些属于 Phase 10 Packaging。配置接口（`processes.json`）设计为 dev/packaged 两种模式可切换，Phase 10 改配置即可零代码改动切到冻结二进制。

This discussion clarified implementation decisions only. Requirements, boundaries, edge coverage, and must-NOT constraints are locked by `07.3-SPEC.md`.

</domain>

## Implementation Decisions

### 日志面板信息架构
- **D-01:** 日志面板复用 Phase 5.3 已建的 [RightPanel](file:///Users/kdocs/Workspace/paladin/apps/desktop/src/components/layout/RightPanel.tsx) 多视图机制，新增一个"日志"视图（与终端/文件预览/Diff 并列）。信息架构一致，代码复用高。
- **D-02:** 日志视图内部两个 tab：`paladin-agent` / `paladin-server`，各自独立流，错误定位清晰。
- **D-03:** v1 日志面板交互 = 实时滚动 + 手动暂停 + 清空，与终端面板能力对齐。搜索/过滤/级别折叠推迟到未来 phase（见 deferred）。
- **D-04:** 落盘日志位置 = Tauri `app_local_data_dir/log/`（macOS: `~/Library/Application Support/com.worsmer.paladin/logs/`，Win: `%LOCALAPPDATA%/com.worsmer.paladin/logs`，Linux: `~/.local/share/com.worsmer.paladin/logs`），跨平台由 Tauri API 自动解析。dev 与 packaged 一致。

### 启动时序与首屏体验
- **D-05:** Agent 状态变 `healthy` 才 mount `CopilotKitProvider`；启动期间显示首屏遮罩。CopilotKit 是 HTTP fetch-per-request 模型，无持久连接健康概念，不会自己重试——所以不能"mount 后让它重试"。
- **D-06:** 遮罩状态驱动颜色：`starting` 蓝、`unhealthy` 黄、`stopped` 红（含退出原因 + 末尾 stderr 摘要 + 手动 restart 按钮）。继承 SPEC P2 prohibition（不静默重启，必须暴露退出原因）。
- **D-07:** Rust 是 URL 唯一真相源：暴露 `get_runtime_config()` Tauri 命令返回 `{agentUrl, serverUrl}`，前端启动时一次 invoke 拿全配置。dev/packaged 统一路径，Rust 控制权完整。**SPEC R8 的 `import.meta.env` 路径被简化为单一 Rust 源**（前端业务代码仍 0 命中 `localhost:9876`，R8 acceptance 满足）。
- **D-08:** 遮罩生命周期与启动对称：手动 `stop_agent` 或运行中变 `stopped`/`unhealthy` 后遮罩重新出现（含对应状态文案 + restart 按钮）。Agent 不 healthy 时遮罩常驻。

### 状态可视化与手动控制 UI
- **D-09:** 现有 [StatusBar.tsx](file:///Users/kdocs/Workspace/paladin/apps/desktop/src/components/StatusBar.tsx) 升级：终端灯旁加 Agent 灯 + Go 灯，3 进程灯一排。`useAgentHealth` 改订阅 Rust event（与 D-07 一致）。**纠正了初版推 ChatToolbar 的建议**——ChatToolbar 在 Agent offline 时随主界面整个不渲染（[App.tsx:167](file:///Users/kdocs/Workspace/paladin/apps/desktop/src/App.tsx#L167)），状态灯放那里在"最需要看状态时"反而看不到。
- **D-10:** [App.tsx:167](file:///Users/kdocs/Workspace/paladin/apps/desktop/src/App.tsx#L167) 现有降级界面升级为 D-06 状态驱动遮罩的载体；StatusBar 始终渲染（无论遮罩/主界面），状态灯永远可见。
- **D-11:** 点状态灯弹 shadcn Popover：含 `restart` / `stop` 按钮 + "查看日志"跳转 RightPanel 日志视图（D-01）+ 末次退出原因（如有，继承 SPEC P2）。
- **D-12:** Go 灯与 Agent 灯并列常驻 StatusBar，不管前端是否直接消费 Go（sidecar 监督职责的一部分 + 为未来 Phase 8 desktop 集成预留）。

### Supervisor 实现路径
- **D-13:** Rust supervisor 自研——`tokio::process` + Tauri Channel + State + Drop 清理，借鉴现有 [TerminalManager](file:///Users/kdocs/Workspace/paladin/apps/desktop/src-tauri/src/terminal/mod.rs) 的模式但代码独立（PTY 是交互式双向、agent/server 是 one-way stdio 捕获，语义不同）。**不引入 `tauri-plugin-shell` 依赖**——它的 `externalBin` 路径偏向冻结二进制，与 dev spawn uv/go run 不完全匹配；探针/退避/状态机仍需自研在它之上，引入它反而分层模糊。PROJECT.md"Tauri 2 sidecar 原生支持"实质为"Tauri 原生进程 API 自研监督"，成色不减。
- **D-14:** 信任 planner 在 SPEC 硬约束内自决剩余技术细节：每进程 task 并发模型（SPEC R7 已锁"每进程监督操作串行化"）、Windows 终止具体机制（`taskkill` / Job Object / `TerminateProcess` 三选一或组合）、重启期间探针策略、ProcessSupervisor 模块组织（独立模块 vs 抽象 trait）。与 Phase 09 CONTEXT 一致（用户信任 planner 在 SPEC 硬约束内自主）。

### Claude's Discretion
- Planner 自选 Rust crate（`tokio` process feature、`reqwest`/`ureq` 探针 HTTP 客户端、日志文件 append 实现）。
- Planner 自选 `processes.json` 字段命名与 schema 形态（数组 vs map、命令用 string vs string array、health 探针配置嵌套层级），只要 dev/packaged 双模式可切换。
- Planner 自选 5 状态的精确色值与文案（中文，与现有"终端运行中/未启动"风格一致），建议用 shadcn design token 体系（Phase 5.1 已迁）而非 raw Tailwind 色。
- Planner 自选 Popover 实现细节（shadcn Popover 已在 Phase 5.1 引入）。
- Planner 自选日志 Channel 协议（event 名、payload schema），只要前端可订阅 + 落盘同步。

### Established Patterns
- **shadcn design system**（Phase 5.1）— UI 组件统一来源；状态灯色值应走 design token 而非 raw Tailwind 色。
- **Tauri Channel + State 模式**（TerminalManager）— 长生命周期子进程 + 流式输出 + 前端订阅的标准范式。
- **配置驱动**（Phase 6 config.json / Phase 8 .env）— `processes.json` 沿用项目"配置文件驱动运行时行为"惯例。
- **固定端口 + env 覆盖**（Phase 8 D-25）— Agent 9876 / Go 9880 固定，端口冲突报错不自动找空闲（SPEC Constraints）。
- **日志绝不含敏感信息**（Phase 8 D-15）— 子进程 stdout/stderr 落盘 + 前端展示必须继承（Agent 的 DEEPSEEK_API_KEY、Go 的 JWT_SECRET/PALADIN_JWT_SECRET 不得出现在日志面板/文件）。

### Integration Points
- `setup()` 钩子（[lib.rs](file:///Users/kdocs/Workspace/paladin/apps/desktop/src-tauri/src/lib.rs)）— 监督器初始化与 spawn 入口；现有 `app.manage(TerminalManager::new(...))` 模式可平行加 `app.manage(ProcessSupervisor::new(...))`。
