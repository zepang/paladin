# Phase 07.3: Sidecar Process Management — Specification

**Created:** 2026-07-05
**Ambiguity score:** 0.11 (gate: ≤ 0.20)
**Requirements:** 9 locked

## Goal

Tauri 桌面端在应用启动时托管 Python Agent 与 Go Server 两个子进程的完整生命周期——启动、健康监测、崩溃自动重启、优雅关闭、日志捕获与暴露、手动控制——使开发者无需再手动 `uv run` / `go run`，且配置驱动以便 Phase 10 切换到冻结二进制时零代码改动。

## Background

当前架构是「Tauri 壳 + 外部独立进程」：

- `tauri.conf.json` 没有 `externalBin`，`src-tauri/Cargo.toml` 没有 `tauri-plugin-shell` / `tokio` process feature，Rust 端**零**子进程 spawn 代码（唯一的 `portable-pty` 仅用于 xterm 终端面板，与 Agent/Server 无关）。
- Python Agent 由开发者手动 `uv run paladin-agent serve --dev`（:9876），Go Server 由开发者手动 `go run ./cmd/server`（:9880）。
- 前端 [App.tsx](file:///Users/kdocs/Workspace/paladin/apps/desktop/src/App.tsx) 三处硬编码 `http://localhost:9876`；[useAgentHealth.ts](file:///Users/kdocs/Workspace/paladin/apps/desktop/src/hooks/useAgentHealth.ts) 仅在组件挂载时 fetch 一次 `/health`，无轮询、无重启、无 Rust 侧监督。
- Tauri 暴露的 7 个命令全部是 terminal PTY + read_text_file，**没有**任何 agent/server 进程命令。

两个子进程自身已具备健康端点：Agent `GET /health`、Go `GET /healthz`（liveness）+ `GET /readyz`（readiness，ping PG+Redis）。Phase 7a 讨论记录已锁定两条继承约束：**Sidecar MVP = spawn + health check + restart**；**崩溃循环退避 1s→2s→4s→8s→16s，最多 5 次**。Phase 8 已完成，SDC-02 的延后条件（"Go Server 存在"）已满足。

本 phase 的核心交付：一个 Rust 监督器模块 + 配置驱动 + 前端订阅，让 Tauri 成为两个子进程的唯一托管者。

## Requirements

1. **子进程启动（SDC-01 + SDC-02 spawn）**：Tauri 在 `setup()` 按 `processes.json` 配置 spawn Python Agent 与 Go Server 子进程，固定端口 9876 / 9880。
   - Current: Tauri 不 spawn 任何子进程；用户手动 `uv run` / `go run`
   - Target: 桌面端启动后两个子进程自动起，端口冲突时报错且不 spawn 该进程
   - Acceptance: 桌面端启动 5s 内两个子进程可见（`ps`/任务管理器）；预先占用 9876 或 9880 时监督器报端口占用错误且不 spawn 该进程、前端显示错误态

2. **健康监测（SDC-03 探针）**：Rust 监督器每 10s 探针 Agent `/health` 与 Go `/healthz` + `/readyz`，状态通过 Tauri event 推前端。
   - Current: 仅前端 useAgentHealth 挂载时 fetch 一次 `/health`
   - Target: Rust 周期探针 + 主动 emit 事件；前端订阅
   - Acceptance: kill 子进程后 ≤30s（10s×3 探针）内前端收到 unhealthy/degraded 事件；前端 useAgentHealth 不再自发 HTTP fetch（改为订阅）

3. **自动重启（SDC-03 重启）**：运行中崩溃（首 5s 后退出）走退避 1/2/4/8/16s 最多 5 次；启动失败（首 5s 内退出）直接 stopped 不退避；Go `/readyz` 失败只报 degraded 不重启。
   - Current: 无任何重启逻辑
   - Target: 区分启动失败 vs 运行崩溃；依赖缺失不误杀
   - Acceptance: kill 子进程后 ≤~40s 内触发重启并恢复 healthy；模拟端口占用导致启动 exit，直接进入 stopped 不走退避（重启间隔不出现 1/2/4/8/16s 序列）；停 PG/Redis，Go 不重启仅状态 degraded

4. **退避耗尽后状态**：5 次退避全失败后进入 stopped，emit stopped 事件，等待手动 restart，不再自动 spawn。
   - Current: N/A
   - Target: 有限重试 + 显式终态
   - Acceptance: 连续制造 5 次启动失败后状态停在 stopped；之后 5 分钟内无自动 spawn；手动 restart 命令可恢复

5. **优雅关闭**：Tauri 退出/崩溃时对每个子进程 SIGTERM（Windows 等价机制）+ 5s grace + 强制 kill，无孤儿残留。
   - Current: Tauri 退出时无子进程清理
   - Target: 可靠的层级关闭
   - Acceptance: 正常关闭桌面端后 `ps` 无残留 paladin-agent / paladin-server；子进程拒绝响应 SIGTERM 时 5s 后被强制 kill（验证进程消失）

6. **日志捕获与暴露**：Rust 捕获子进程 stdout/stderr，经 Tauri Channel 流推前端日志面板，同时追加落盘到 OS 适当的 logs 目录。
   - Current: 子进程输出到各自终端，桌面端不可见
   - Target: 前端可见 + 文件落盘
   - Acceptance: 子进程输出一条含特征字符串的日志，前端日志面板 1s 内可见；logs 目录下 `paladin-agent.log` / `paladin-server.log` 各自包含该字符串

7. **手动控制命令**：暴露 `restart_agent` / `stop_agent` / `restart_server` / `stop_server` / `get_process_status` 五个 Tauri 命令。
   - Current: 无任何 agent/server Tauri 命令
   - Target: 前端可手动控制
   - Acceptance: invoke `restart_agent` 后 Agent 进程 PID 改变；`stop_agent` 后进程消失；`get_process_status` 返回 `{agent, server}` 各自状态（starting/running/degraded/unhealthy/stopped）

8. **配置驱动 URL**：Agent URL 从硬编码迁到 `import.meta.env`（dev）+ 运行时配置（packaged），前端三处统一从单一配置读。
   - Current: 前端 [App.tsx](file:///Users/kdocs/Workspace/paladin/apps/desktop/src/App.tsx) 三处硬编码 `http://localhost:9876`
   - Target: 单一配置源
   - Acceptance: `grep -rn "localhost:9876" apps/desktop/src` 返回 0 业务代码命中（仅配置文件/默认值常量）；改配置后前端指向新地址

9. **跨平台**：监督逻辑在 macOS、Windows、Linux 三平台均工作（spawn、信号、路径、日志目录）。
   - Current: 仅 macOS 实测（dev 手动启动）
   - Target: 三平台一致
   - Acceptance: 三平台各执行一遍启动 + 健康 + 重启 + 关闭 UAT 通过

## Boundaries

**In scope:**
- Rust 监督器模块（spawn / 健康探针 / 重启 / 退避 / 优雅关闭 / 日志捕获）
- `processes.json` 配置 schema + 加载与校验
- 5 个 Tauri 命令 + 日志 Tauri Channel
- 前端 `useAgentHealth` 从自发 fetch 改为订阅 Rust 事件
- 前端 URL 迁配置驱动（`import.meta.env` + 运行时配置）
- 前端日志面板（订阅子进程输出）
- 日志文件落盘（OS 适当目录）
- 三平台（macOS / Windows / Linux）适配

**Out of scope:**
- Python Agent 二进制冻结（PyInstaller / Nuitka）→ Phase 10 Packaging，Pydantic AI 冻结风险独立
- Go 二进制按 target-triple 命名构建矩阵 → Phase 10 Packaging
- Tauri `externalBin` 配置 + `.dmg` / `.msi` 打包 → Phase 10
- Postgres / Redis 拉起（docker-compose 编排）→ 用户/运维自带；Go `/readyz` 间接反映依赖可用性
- CopilotKit Runtime Node 中间层 → 现有直连 AG-UI 架构不变
- 远程 agent / 多机器分发 → 未来版本，v1 仅本地子进程
- 高可用集群 / 性能压测 → 非本 phase 目标
- 日志轮转 / 大小上限 / 压缩 → Phase 10 ops（v1 已知限制：无界追加）

## Constraints

- **继承自 Phase 7a（锁定，不可改）**：崩溃重启退避序列 1s → 2s → 4s → 8s → 16s，最多 5 次。
- **端口**：Agent 9876 / Go 9880 固定，写死在 `processes.json`；占用时报错，不自动寻找空闲端口。
- **探针节奏**：每 10s 一次；连续 3 次失败触发重启。
- **启动失败窗口**：首 5s 内 exit 视为启动失败（边界含 5s），直接 stopped 不退避。
- **优雅关闭 grace**：5s，超时强制 kill。
- **健康端点协议**：Agent `/health`；Go `/healthz`（liveness，触发重启）+ `/readyz`（readiness，PG/Redis 挂时只报 degraded）。
- **配置驱动接口**：`processes.json` schema 必须支持 dev 模式（`cmd: "uv run ..."`）与 packaged 模式（`cmd: "<binary path>"`）两种，让 Phase 10 切换零代码改动。
- **跨平台**：macOS / Windows / Linux 全部；Windows 无 SIGTERM，需用等价终止机制。
- **可借鉴模式**：现有 [TerminalManager](file:///Users/kdocs/Workspace/paladin/apps/desktop/src-tauri/src/terminal/mod.rs)（Tauri Channel + State + Drop 清理）是监督器的实现范本。

## Acceptance Criteria

核心功能：
- [ ] 桌面端启动后 5s 内 Agent + Go Server 两个子进程自动 spawn，PID 可查
- [ ] 端口 9876 或 9880 被占时，监督器报错且不 spawn 该进程，前端显示错误态
- [ ] kill 子进程后 ≤30s 内监督器检测并触发重启（10s×3 探针）
- [ ] 重启退避间隔依次为 1s / 2s / 4s / 8s / 16s
- [ ] 连续 5 次重启失败后状态停在 stopped，之后 5 分钟内无自动 spawn
- [ ] 启动首 5s 内 exit 视为启动失败，直接 stopped 不走退避
- [ ] 停 PG/Redis，Go Server 不重启，仅状态变 degraded
- [ ] Agent 端点非 200 或 body 异常计为一次探针失败（计入 3 连败阈值）
- [ ] 退出时刻 t≤5s 归启动失败；任何终止原因（非零退出 / 信号 / 崩溃）均算
- [ ] 正常关闭桌面端后 `ps` 无残留 paladin-agent / paladin-server
- [ ] SIGTERM 无响应时 5s 后强制 kill 验证

日志与控制：
- [ ] 子进程 stdout/stderr 在前端日志面板 1s 内可见，并落盘 logs 目录
- [ ] logs 目录不可写时退化为仅 stderr + 发 warning 事件，不阻塞 spawn
- [ ] `restart_agent` / `stop_agent` / `restart_server` / `stop_server` / `get_process_status` 命令工作
- [ ] `get_process_status` 在首次探针完成前返回 `starting`
- [ ] 并发 restart 同一进程返回 in-progress，不二次 spawn
- [ ] 手动 restart 命令把该进程失败计数清零

配置与跨平台：
- [ ] `grep -rn "localhost:9876" apps/desktop/src` 业务代码 0 命中
- [ ] URL 配置缺失时回退默认 http://localhost:9876 + console 警告
- [ ] `processes.json` 缺失/格式错时监督器报配置错误、不 spawn、前端显示配置错误态
- [ ] macOS、Windows、Linux 三平台启动 + 健康 + 重启 + 关闭 UAT 通过
