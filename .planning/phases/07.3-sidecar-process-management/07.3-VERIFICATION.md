## §2. SPEC 4 条 Prohibitions 逐条验证

### P1. 不引入命令注入风险 — test asserts cmd 来自 static config only

- **状态:** `PASS`
- **验证方法:** type system 锁 + tokio::process no-shell + 3 held-out 单元测试
- **证据:**
  - [config.rs](file:///Users/Workspace/paladin/apps/desktop/src-tauri/src/process/config.rs) `ProcessConfig::load_from_path(impl AsRef<Path>)` —— `AsRef<Path>` 编译期排除 `reqwest::Url` 与 Tauri IPC 字符串
  - 测试 `_assert_signature_accepts<T: AsRef<Path>>` 编译期证明(plan 03 SUMMARY)
  - `supervisor.rs` 用 `tokio::process::Command::new(&entry.cmd[0]).args(&entry.cmd[1..])` —— **无 shell**(`/bin/sh -c` 调用),参数 array 直传内核 execve
  - cmd 来源仅 [processes.json](file:///Users/kdocs/Workspace/paladin/apps/desktop/src-tauri/processes.json),启动时一次性 load,运行时不可变
  - 3 个「held-out」测试(plan 03 SUMMARY §Prohibition guards):`test_rejects_url_input` / `test_rejects_string_input` / `test_rejects_user_ipc_input`

### P2. 不静默重启 — UI 含退出原因 + stderr

- **状态:** `PASS`
- **验证方法:** emit_status_to exposes last_error + stderr_tail + UI 组件渲染
- **证据:**
  - `supervisor.rs` `emit_status_to` 在状态变化时附带 `last_error`(中文消息)+ `stderr_tail`(最近 N 行 stderr)
  - 前端 [StartupMask.tsx](file:///Users/kdocs/Workspace/paladin/apps/desktop/src/components/StartupMask.tsx) 渲染 last_error(中文)+ stderr_tail(monospace)
  - 前端 [ProcessLight.tsx](file:///Users/kdocs/Workspace/paladin/apps/desktop/src/components/ProcessLight.tsx) 在 Degraded/Unhealthy 时 tooltip 显示 last_error
  - plan 06 SUMMARY §Accomplishments 7:「UI 含退出原因 + stderr(SPEC Prohibition P2)」

### P3. stopped 不自恢复 — test stopped no auto-spawn within 5min

- **状态:** `PASS`
- **验证方法:** Stopped 终态无 timer + state_machine 测试三件套
- **证据:**
  - state_machine 测试 `test_stopped_ignores_backoff_tick` / `test_stopped_ignores_probe_fail` / `test_stopped_ignores_probe_ok`:Stopped 对 BackoffTick/ProbeFail/ProbeOk 全免疫
  - `supervisor.rs` 状态 = Stopped 时 `probe_loop` 与 `wait_exit` 都 `break`,**无任何 tokio::time::interval / sleep** 唤醒
  - 唯一脱离路径:`RestartRequested`(手动 invoke) — 用户主动行为不算「自恢复」
  - plan 02 SUMMARY §Prohibition guards P3

### P4. 不裸 PID kill — test PID 复用不杀哨兵

- **状态:** `PASS`
- **验证方法:** 只持有 tracked Child handle,无 PID file,无 system kill by PID
- **证据:**
  - `supervisor.rs` `ManagedProcess.child: Option<tokio::process::Child>` —— 唯一持有 Child handle
  - 关闭走 `child.start_kill()`(Win)/ `child.kill()`(Unix 强杀兜底)/ `nix::sys::signal::kill(child_pid, SIGTERM)`(Unix grace)
  - nix kill 用的是 `Child::id()` 返回的 pid,**但**仅在 `child.is_some()` 时调用 —— 即只在 handle 仍存活时
  - **不**写 PID file,**不**读外部 pid 文件,**不**接受外部 pid invoke 参数 → PID 复用场景下 OS 不会把外部进程当 paladin 子进程杀
  - plan 05 SUMMARY §Decisions D-14:「Windows Job Object 而非 PID」

**Prohibitions 统计:4/4 PASS。**
