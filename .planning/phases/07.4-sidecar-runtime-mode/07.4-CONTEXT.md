# Phase 07.4: sidecar-runtime-mode - Context

**Gathered:** 2026-07-06
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 07.4 implements the runtime-mode contract locked by SPEC: Paladin's Tauri process supervisor must distinguish service health from process ownership. In dev mode it attaches healthy already-running Agent/Go services as external, spawns absent services on free ports as supervisor-owned, and reports occupied unhealthy ports as conflict. In packaged mode it preserves a supervisor-owned, dev-tool-free path without building real frozen binaries.

This discussion clarified implementation decisions only. Requirements, boundaries, edge coverage, and must-NOT constraints are locked by `07.4-SPEC.md`.

</domain>

- Tauri `externalBin`, `.dmg`, `.msi`, or cross-platform installer validation - Phase 10 owns packaging.
- Persisting a user preference for external vs managed startup - dev attach is session-only in this phase.
- A one-click "switch to managed" flow that kills or stops external processes - users stop external services themselves, then redetect.
- Starting PostgreSQL or Redis from Tauri - dependencies remain external; Go `/readyz` reports readiness degradation.
- Remote/multi-machine Agent or Server support - v1 remains local.
- Log rotation/compression and long-running ops hardening - already deferred to Phase 10 ops.

<decisions>
## Implementation Decisions

### Runtime Mode Configuration Boundary
- **D-01:** `mode=packaged` validation is strict. If packaged config uses dev commands such as `uv` or `go`, or repo-relative dev cwd such as `../../agent` / `../../server`, config validation fails, supervisor management does not start, and the UI surfaces a diagnostic configuration error.
- **D-02:** `augmented_spawn_path()` is dev-only. It may run only in `mode=dev` when spawning a supervisor-owned child. The packaged branch MUST NOT call PATH augmentation.
- **D-03:** Keep the existing `cmd: string[]` command schema in this phase. Do not split into `dev_cmd` / `packaged_cmd` or introduce a new `binary` field; enforce packaged-mode safety with validation and runtime branch tests.
- **D-04:** A `processes.json` parse/schema/validation failure is global for process supervision. It blocks supervisor management for both services and should be visible in the desktop UI rather than only stderr/logs. Do not implement per-service config isolation in this phase.

### External Attach Health and Redetect Semantics
- **D-05:** After a service attaches as `owner=external`, Rust supervisor still probes its configured liveness/readiness endpoints and emits status updates. Probe failures for external services MUST NOT trigger restart, kill, or graceful shutdown.
- **D-06:** If a previously attached external service later fails health, represent it as `state=unhealthy`, `owner=external`, `health=failed`. Reserve `state=conflict`, `owner=none`, `health=failed` for preflight/redetect cases where a port is occupied and the configured health endpoint does not pass.
- **D-07:** Add explicit Tauri commands `redetect_agent` and `redetect_server`. `restart_*` MUST NOT be reused for redetect because restart remains a supervisor-owned action and external restart is prohibited.
- **D-08:** `redetect_*` may switch from external to supervisor-owned only when the user has manually stopped the external service and the configured port is now free. In that case, redetect follows the dev hybrid rule and spawns a supervisor-owned child. No confirmation dialog is required.

### UI Copy and Control Menu Shape
- **D-09:** In `ProcessLight`, keep stop/restart controls visible but disabled for `owner=external`, with explanatory copy such as "外部服务由你手动管理，Paladin 不会停止或重启它". Show "重新检测" as the available action.
- **D-10:** StatusBar/ProcessLight labels should show both ownership and state in a compact form, for example `Agent · 外部 · 运行中`, `Go · 托管 · 降级`, and `Agent · 冲突`.
- **D-11:** `StartupMask` is driven only by Agent availability. Go Server degraded/conflict/unhealthy status is shown through StatusBar/Popover diagnostics and does not block the CopilotKit workspace.
- **D-12:** Failure-class diagnostics use short title + diagnostic block + suggested action, following the current `StartupMask` diagnostic block style. Do not add a large runbook/help system in this phase.

### State Model and Concurrency Ownership Boundary
- **D-13:** Define a legal `state/owner/health` combination table and test it. The implementation should prevent obviously invalid tuples such as `conflict + supervisor + healthy`. Expected valid examples include `running/external/healthy`, `running/supervisor/healthy`, `degraded/external|supervisor/degraded`, `unhealthy/external/failed`, `conflict/none/failed`, and stopped states with `owner=none|supervisor` plus `health=unknown|failed`.
- **D-14:** `stop_*` and `restart_*` encountering `owner=external` return a typed error, for example `external_not_owned`, and perform no kill, restart, spawn, or redetect action. UI disabling is a convenience layer, not the safety boundary.
- **D-15:** Graceful shutdown and app quit only stop supervisor-owned tracked child handles. `owner=external` and `owner=none` are skipped and must never be killed by PID matching or stale process references.
- **D-16:** Use one per-service runtime lock that covers preflight, redetect, spawn, restart, and stop. Do not leave redetect/start outside the existing restart serialization; avoid a global supervisor lock that unnecessarily blocks Agent and Go operations from each other.

### Verification Strategy and Test Doubles
- **D-17:** Dev attach/conflict tests should use a local fake HTTP server with ephemeral ports to simulate health 200, health failure, connection refused, and occupied ports. This proves the real port + HTTP behavior without requiring real Agent/Go services.
- **D-18:** Packaged no-dev-deps requires both config validation tests and spawn-path branch tests. Tests must prove packaged mode rejects dev commands/repo cwd and does not call PATH augmentation.
- **D-19:** Frontend coverage should combine TypeScript/build checks with lightweight component behavior tests or equivalent focused tests for new unions, disabled external controls, conflict labels, and the rule that Go status does not block `StartupMask`.
- **D-20:** Final gates should include Rust process/runtime tests, frontend type/build checks, and focused source gates for packaged no-dev-deps and external stop/restart disabling. Full manual UAT remains secondary and can be deferred to Phase 10 where packaging/UAT already lives.

### the agent's Discretion
- Planner may choose exact Rust module/function names for preflight/redetect classification, as long as the legal tuple table and lock boundary above are preserved.
- Planner may choose the exact typed error representation for `external_not_owned` and config errors, as long as frontend can render user-facing Chinese diagnostics.
- Planner may choose the lightweight frontend test mechanism already most natural for this repo, but it must prove behavior rather than only compile types.
- Planner may refine exact Chinese copy, icons, and colors within the established shadcn/lucide style and compact StatusBar pattern.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `ProcessSupervisor` already owns per-service slots, child handles, probe loops, restart logic, shutdown logic, Tauri event emission, and runtime config.
- `ProcessConfig::load_from_path` and `validate()` already provide the right place for strict mode-specific config checks.
- `port_probe::is_port_available` / `check_port_or_error` already prove port occupancy, but must move behind health-first preflight in dev mode.
- `ProcessLight` already has Popover actions for restart/stop/logs; extend it instead of creating a new control surface.
- `StartupMask` already has shadcn-style diagnostic blocks with title, description, error, stderr tail, and action.
- Existing Rust tests cover config, port probing, state machine, log redaction, and PATH augmentation; extend this test family for 07.4.

### Established Patterns
- Rust remains the runtime truth source. Do not move external health polling to the frontend.
- UI should extend existing StatusBar/ProcessLight/StartupMask patterns and use shadcn/lucide conventions.
- The project prefers compact, diagnostic operational UI over new large panels for every state.
- Sidecar lifecycle actions must be conservative: only tracked child handles may be killed.
- Dev/packaged duality should be configuration-driven, preserving Phase 10's ability to swap in real packaged binaries later.

### Integration Points
- `setup()` currently loads config and starts supervisor in the background; config error handling currently returns setup error, but Phase 07.4 needs a user-visible configuration diagnostic.
- `spawn_child_to_slot` currently checks port availability before spawn; Phase 07.4 introduces a preflight classifier before spawn.
- `probe_loop` currently transitions running/degraded/unhealthy and can trigger supervisor-owned restart via existing crash/probe logic; external owner must update health without restart.
- `graceful_shutdown_slot` currently acts on child handles; Phase 07.4 must guard it by ownership and preserve the tracked-child-only invariant.
- Frontend `ProcessState` currently lacks `conflict`; `ProcessInfo` lacks `owner` and `health`.

</code_context>

<specifics>
## Specific Ideas

- The user selected every gray area for discussion and consistently chose the stricter/safest implementation options.
- The preferred UX is explicit and diagnostic: disabled external controls should explain why, conflict should show port/health diagnostics, and failure classes should include a suggested action.
- The preferred verification style is automated-first with real behavior where cheap: fake HTTP server + ephemeral ports for attach/conflict, plus source gates for no-dev-deps and external-action safety.
- The user does not want a persisted external/managed preference or one-click switch-to-managed flow in this phase.

</specifics>
