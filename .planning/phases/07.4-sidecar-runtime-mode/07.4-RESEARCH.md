## User Constraints (from CONTEXT.md)

### Locked Requirements and Boundaries
- Phase 07.4 implements the runtime-mode contract locked by SPEC: distinguish service health from process ownership. In dev mode attach healthy already-running Agent/Go services as external, spawn absent services on free ports as supervisor-owned, and report occupied unhealthy ports as conflict. In packaged mode preserve a supervisor-owned, dev-tool-free path without building real frozen binaries. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]
- Downstream agents must read `07.4-SPEC.md`; requirements are not duplicated as implementation discretion. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]
- In scope: `ProcessState` / DTO / frontend store changes for `state + owner + health`; dev hybrid preflight; packaged runtime-mode branch/contract; redetect behavior; UI labels/actions; Rust automated tests and frontend type/component checks. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]
- Out of scope: real frozen Python/Go binaries, Tauri `externalBin`, installers, persisted external/managed preference, one-click switch-to-managed that kills external processes, Tauri-started PostgreSQL/Redis, remote/multi-machine support, log rotation/compression and long-running ops hardening. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]

### Locked Decisions
- `mode=packaged` validation must be strict: dev commands such as `uv` or `go`, and repo-relative dev cwd such as `../../agent` / `../../server`, must fail validation and surface a desktop diagnostic. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]
- `augmented_spawn_path()` is dev-only and must not run in packaged mode. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]
- Keep the existing `cmd: string[]` schema; do not split into `dev_cmd` / `packaged_cmd` or add a `binary` field in this phase. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]
- A `processes.json` parse/schema/validation failure is global for supervision and should be visible in the desktop UI, not only stderr/logs. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]
- After `owner=external` attach, Rust supervisor still probes liveness/readiness and emits status updates; probe failures must not trigger restart, kill, or graceful shutdown. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]
- External health failure after attach is `state=unhealthy`, `owner=external`, `health=failed`; preflight/redetect occupied unhealthy port is `state=conflict`, `owner=none`, `health=failed`. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]
- Add explicit Tauri commands `redetect_agent` and `redetect_server`; do not reuse `restart_*` for redetect. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]
- `redetect_*` may switch from external to supervisor-owned only after the user manually stops the external service and the port is free. No confirmation dialog required. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]
- In `ProcessLight`, stop/restart remain visible but disabled for `owner=external` with explanatory copy; "重新检测" is the available action. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]
- Status labels should compactly show ownership and state, e.g. `Agent · 外部 · 运行中`, `Go · 托管 · 降级`, `Agent · 冲突`. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]
- `StartupMask` is driven only by Agent availability. Go degraded/conflict/unhealthy is shown through StatusBar/Popover diagnostics and must not block the CopilotKit workspace. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]
- Failure-class diagnostics use short title + diagnostic block + suggested action; do not add a large runbook/help system. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]
- Define and test a legal `state/owner/health` combination table. Invalid examples such as `conflict + supervisor + healthy` must be prevented. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]
- `stop_*` and `restart_*` on `owner=external` return a typed error such as `external_not_owned` and perform no kill, restart, spawn, or redetect. UI disabling is not the safety boundary. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]
- Graceful shutdown and app quit only stop supervisor-owned tracked child handles; never kill external services by PID matching or stale process references. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]
- Use one per-service runtime lock covering preflight, redetect, spawn, restart, and stop; avoid a global supervisor lock. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]
- Dev attach/conflict tests should use a local fake HTTP server with ephemeral ports. Packaged tests must prove packaged mode rejects dev commands/repo cwd and does not call PATH augmentation. Frontend tests should prove behavior, not only compile types. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]

## Summary

Plan 07.4 as a refactor of the existing Phase 07.3 Rust supervisor, not as a new process manager. The current implementation already loads `processes.json`, spawns Server then Agent, runs reqwest probes, captures logs, emits `process-status`, exposes Tauri commands, and drives `StartupMask`/`StatusBar`/`ProcessLight`. [VERIFIED: codebase]

The missing abstraction is a runtime classification layer before spawn and before lifecycle actions: `state` must stop carrying both health and ownership semantics. Add `owner` and `health` to Rust `ManagedProcess`, `ProcessInfoDTO`, event payloads, TypeScript store types, and UI render paths. Add `state=conflict`. [VERIFIED: codebase]

The most important sequencing change is dev preflight: probe liveness first, then decide attach/spawn/conflict. The current `spawn_child_to_slot` still calls `check_port_or_error` before any health probe, so a healthy manually started service is treated as a spawn blocker. [VERIFIED: codebase]

## Existing Implementation Map

- `apps/desktop/src-tauri/processes.json` is current dev config: `mode=dev`, Agent command `["uv","run","paladin-agent","serve","--dev"]`, Agent port `9876`, Agent liveness `/health`, Server command `["go","run","./cmd/server"]`, Server port `9880`, Server liveness `/healthz`, Server readiness `/readyz`. [VERIFIED: codebase]
- `ProcessConfig` already has `mode: String`, `processes`, `backoff_secs`, `max_restarts`, and `shutdown_grace_secs`; validation currently checks generic ranges and endpoint status codes but not allowed modes, required agent/server entries, cwd existence, packaged dev-tool prohibition, or repo-relative packaged cwd prohibition. [VERIFIED: codebase]
- `ProcessState` currently has only `starting | running | degraded | unhealthy | stopped`; no `conflict`, no owner enum, no health enum. [VERIFIED: codebase]
- `ProcessInfoDTO` currently exposes `state`, `last_error`, `stderr_tail`, `last_restart_at`; `process-status` flattens this DTO with `name`. [VERIFIED: codebase]
- `ManagedProcess` currently stores `state`, `snapshot`, `stderr_tail`, `spawn_time`, `last_restart_at`, and `last_error`; no child ownership marker exists. [VERIFIED: codebase]
- `ProcessSlot` already has per-service `restart_lock`; use or rename it as the per-service runtime lock required by 07.4. [VERIFIED: codebase]
- `ProcessSupervisor::start()` currently always spawns Server first, waits up to 5s for Server `/readyz`, marks Server degraded if not ready, then spawns Agent. It does not attach external services. [VERIFIED: codebase]
- `spawn_child_to_slot()` currently checks port availability before spawn, always uses `augmented_spawn_path(env::var_os("PATH"))`, sets state `Starting`, stores a tracked `Child`, starts stdout/stderr capture, `probe_loop`, and `wait_exit`. [VERIFIED: codebase]
- `probe_loop()` probes liveness every 10s after an initial 10s sleep; liveness OK plus readiness OK becomes `ProbeOk`, liveness OK plus readiness fail becomes `ProbeDegraded`, liveness fail becomes `ProbeFail`. [VERIFIED: codebase]
- `wait_exit()` assumes there is a tracked child and maps <=5s exit to `SpawnFailed -> Stopped`; >5s exit to `Crashed -> Unhealthy` plus backoff respawn. External services must be excluded from this path. [VERIFIED: codebase]
- `graceful_shutdown_slot()` only acts on tracked child handles and does not PID-match; preserve this invariant and add an ownership guard before state mutation/kill. [VERIFIED: codebase]
- `lib.rs` currently returns a setup error if `processes.json` load fails, which can prevent UI diagnostics. 07.4 requires a visible configuration diagnostic, so planner must decide whether to manage a supervisor-like error state or a separate config status resource. [VERIFIED: codebase]
- Frontend `ProcessState` union lacks `conflict`; `ProcessInfo` lacks owner/health; `initProcessListeners()` merges partial event patches by process name. [VERIFIED: codebase]
- `App.tsx` blocks CopilotKitProvider whenever Agent state is not `running`; it does not inspect Go status. This aligns with D-11 and should be preserved with `conflict/unhealthy/external` additions. [VERIFIED: codebase]

## Architecture Patterns

- Keep command execution shell-free: `Command::new(&entry.cmd[0]).args(&entry.cmd[1..])` is the established security boundary. Preserve `cmd: Vec<String>`. [VERIFIED: codebase]
- Keep runtime URLs derived from Rust config through `get_runtime_config`; frontend has only fallback defaults for config-load failure. [VERIFIED: codebase]
- Keep log behavior centralized: `capture_lines()` redacts before emitting `process-log` and preserves stderr tail for diagnostics. 07.4 failure classes should reuse `last_error`/`stderr_tail` rather than introduce a second log channel. [VERIFIED: codebase]
- Keep readiness separate from liveness: Go `/readyz` degraded does not count as liveness failure and does not restart. [VERIFIED: codebase]
- Prefer pure-function tests for tuple legality and classifier behavior, mirroring `state_machine.rs` and `port_probe.rs`. [VERIFIED: codebase]
- Preserve compact operational UI: `StatusBar` hosts two `ProcessLight`s; `StartupMask` is a centered diagnostic panel; avoid new large panels/runbooks. [VERIFIED: codebase]

## Don't Hand-Roll

- Do not build a new process supervisor; extend `ProcessSupervisor`, `ProcessSlot`, `ManagedProcess`, and existing command wiring. [VERIFIED: codebase]
- Do not implement custom shell parsing or single-string commands; `cmd: string[]` and `Command::new(...).args(...)` already solve command injection risk. [VERIFIED: codebase]
- Do not add frontend-side health polling; Rust is the truth source and already owns reqwest probes. [VERIFIED: codebase]
- Do not kill by port, PID file, process name, or system process search. Only tracked `Child` handles with `owner=supervisor` may be stopped. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-SPEC.md]
- Do not introduce persisted external/managed preferences or a kill-based switch-to-managed flow. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-SPEC.md]
- Do not make packaged mode depend on `uv`, `go`, login-shell `PATH`, or repo-relative cwd. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-SPEC.md]

## Runtime State Inventory

Current status tuple:
- Rust: `ProcessState = Starting | Running | Degraded | Unhealthy | Stopped`. [VERIFIED: codebase]
- TypeScript: `ProcessState = 'starting' | 'running' | 'degraded' | 'unhealthy' | 'stopped'`. [VERIFIED: codebase]
- DTO/store fields: `state`, `last_error`, `stderr_tail`, `last_restart_at`. [VERIFIED: codebase]

Target tuple for planner:
- `state`: `starting | running | degraded | unhealthy | stopped | conflict`. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-SPEC.md]
- `owner`: `supervisor | external | none`. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-SPEC.md]
- `health`: `healthy | degraded | failed | unknown`. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-SPEC.md]

Legal tuple table to encode and test:
- `starting/supervisor/unknown`: supervisor child spawned; first probe pending. [ASSUMED]
- `running/supervisor/healthy`: managed child liveness OK and readiness OK or absent. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]
- `running/external/healthy`: dev preflight or redetect attached a healthy existing service. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]
- `degraded/supervisor/degraded`: managed Go liveness OK but readiness failed. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]
- `degraded/external/degraded`: external Go liveness OK but readiness failed. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]
- `unhealthy/supervisor/failed`: managed child probe failure threshold or crash/backoff path. [VERIFIED: codebase]
- `unhealthy/external/failed`: previously attached external service later fails liveness. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]
- `stopped/supervisor/unknown|failed`: managed child stopped manually, spawn failed, or backoff exhausted. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]
- `stopped/none/unknown`: no service running and no supervisor-owned child attached/spawned. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]
- `conflict/none/failed`: port occupied and configured health endpoint failed during preflight/redetect. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]

Invalid tuples planner must reject or make unrepresentable:
- `conflict/supervisor/*`.
- `conflict/external/*`.
- `conflict/none/healthy`.
- `running/none/*`.
- `starting/external/*`.
- `degraded/none/*`.
- `unhealthy/none/*` unless planner deliberately models global config failure separately; prefer config diagnostic outside per-service ownership. [ASSUMED]

## Common Pitfalls

- Treating a port bind failure as conflict before trying liveness will break healthy external attach. Health-first is the central behavioral change. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-SPEC.md]
- Reusing `restart_*` for redetect will violate the locked command semantics. Add `redetect_*`. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]
- Allowing `graceful_shutdown_slot()` to set external service state to stopped before discovering no child handle would misrepresent external services. Guard by owner before transition. [VERIFIED: codebase]
- Letting `probe_loop()` for external services call transition paths that increment supervisor restart counts or trigger restart would violate ownership separation. [VERIFIED: codebase]
- Keeping `augmented_spawn_path()` unconditional in `spawn_child_to_slot()` would violate packaged mode. Branch on `config.mode` and test packaged does not call it. [VERIFIED: codebase]
- Config load failures currently abort Tauri setup. That prevents the UI from rendering a diagnostics state, contrary to D-04. [VERIFIED: codebase]
- StartupMask currently blocks on any Agent state other than `running`. If `running/external/healthy` is represented only by `owner` and not `state=running`, the workspace will remain blocked. [VERIFIED: codebase]
- Go degraded/conflict should not block workspace because `App.tsx` only consults Agent status. Preserve this boundary. [VERIFIED: codebase]

## Edge Coverage and Prohibitions for Planner

Lift these SPEC items into `must_haves`:
- Repeated identical status updates must leave `(state, owner, health)` stable. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-SPEC.md]
- Concurrent event delivery must not collapse owner into state. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-SPEC.md]
- Dev preflight/redetect must be per-service serialized; repeated redetect on healthy external remains external; concurrent calls cannot double-spawn. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-SPEC.md]
- Packaged startup remains supervisor-owned and single-child; repeated startup must not reintroduce dev command dependencies. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-SPEC.md]
- External stop/restart clicks are no-ops/rejections; repeated redetect reclassifies only from current health/port state. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-SPEC.md]
- Verification matrix must include dev attach, dev spawn, conflict, packaged supervisor-owned, external action disabled, and failure classes; missing any category fails verification. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-SPEC.md]
- `owner=external` services must not be killed, stopped, restarted, gracefully shut down, or supervisor-recovered. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-SPEC.md]
- Healthy external Agent/Go must not be reported stopped/unhealthy/conflict merely because it is not a supervisor child. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-SPEC.md]
- Packaged mode must not start through `uv`, `go`, login-shell `PATH`, or repo-relative dev cwd. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-SPEC.md]
- Occupied port with failed configured health must surface as `conflict` with port and health diagnostics, not generic stopped/startup failure. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-SPEC.md]

## Validation Architecture

Rust validation:
- Add pure tuple legality tests for every valid/invalid `state/owner/health` combination.
- Add preflight classifier tests using fake HTTP servers and ephemeral ports:
  - liveness 200 -> `running/external/healthy`, no spawn.
  - liveness connection refused + port free -> spawn path selected.
  - occupied port + liveness non-expected/timeout -> `conflict/none/failed`, no spawn.
  - Go liveness 200 + readiness 503 -> `degraded/{external|supervisor}/degraded`.
- Add concurrency tests around per-service runtime lock: repeated/concurrent start/redetect/restart cannot double-spawn or flip healthy external into managed.
- Add command tests: `stop_*` and `restart_*` return typed `external_not_owned` for external and perform no child kill/spawn.
- Add packaged config validation tests: packaged rejects `uv`, `go`, `../../agent`, `../../server`; packaged accepts non-dev configured binary shape; packaged branch does not call `augmented_spawn_path()`.

Frontend validation:
- Add TypeScript/type checks for `ProcessState`, `ProcessOwner`, `ProcessHealth`, and event payload/store merges.
- Add focused Vitest/component tests for:
  - external controls disabled with explanatory copy and redetect enabled.
  - conflict labels/diagnostics include port/health failure.
  - Go degraded/conflict does not trigger `StartupMask`; Agent non-running still does.
  - failure classes render distinct titles/suggested actions for executable missing, cwd missing, port conflict, health failure, startup-grace exit, readiness degraded.

Suggested final gates:
- `cd apps/desktop/src-tauri && cargo test`
- `cd apps/desktop/src-tauri && cargo clippy --all-targets -- -D warnings`
- `pnpm --filter @paladin/desktop build`
- `pnpm --filter @paladin/desktop test -- --run` if component tests are added.
- Optional unchanged-service regression: `cd apps/server && go test ./... -race -count=1`; `cd apps/agent && uv run pytest -v` if health endpoint behavior is touched.

## Security Domain

- Command injection boundary already exists through `cmd: Vec<String>` and no shell invocation. Keep this invariant while adding packaged validation. [VERIFIED: codebase]
- External ownership is a safety boundary: Rust commands must reject external stop/restart, not only disable UI controls. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-CONTEXT.md]
- Shutdown safety depends on tracked child handles only; do not add PID scanning, port-kill, or process-name kill behavior. [CITED: .planning/phases/07.4-sidecar-runtime-mode/07.4-SPEC.md]
- `processes.json` remains a local filesystem config loaded from `CARGO_MANIFEST_DIR`; avoid remote or IPC-provided command config in this phase. [VERIFIED: codebase]
- Logs are already redacted before emit/file append. Do not bypass `redact_log_line()` in any new diagnostics path. [VERIFIED: codebase]
- Packaged mode validation is security-relevant because a packaged app relying on developer PATH or repo cwd can execute unexpected tools. [ASSUMED]

## Assumptions Log

- The planner can choose whether tuple legality lives inside an extended `state_machine.rs` or a new pure module such as `runtime_mode.rs`, as long as async supervisor code consumes a tested pure decision layer. [ASSUMED]
- A small local fake HTTP server in Rust tests is acceptable even though no helper exists yet; `reqwest` is already a dependency and `std::net::TcpListener` is already used in tests. [ASSUMED]
- "Executable missing" can be represented from `Command::spawn()` errors, while "cwd missing" should be caught earlier or classified from `current_dir`/spawn error text; planner should prefer explicit cwd existence validation for clearer diagnostics. [ASSUMED]
- Config diagnostic UI may require a supervisor/global status object because current setup aborts before the React app can render. [ASSUMED]

## Open Questions

- Should `ProcessConfig::mode` become a typed enum (`RuntimeMode`) rather than `String`? Strong recommendation: yes, because 07.4 branches on mode and invalid mode should be a schema error. [ASSUMED]
- Should config load failure be modeled as per-service `stopped/none/failed` or a separate global `process_config_error` status? Strong recommendation: separate global diagnostic if feasible, to avoid inventing invalid per-service ownership. [ASSUMED]
- Should Agent ever enter `degraded`? Current Agent has no readiness endpoint, so probably no; keep the union but tests should focus Agent degraded only if future config includes readiness. [VERIFIED: codebase]
- Should external probe loops start immediately after attach rather than waiting 10s? Recommendation: run an immediate preflight result emit, then continue with the existing 10s cadence. [ASSUMED]
