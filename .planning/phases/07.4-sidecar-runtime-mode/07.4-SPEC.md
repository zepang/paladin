# Phase 07.4: Sidecar Runtime Mode - Specification

**Created:** 2026-07-06
**Ambiguity score:** 0.15 (gate: <= 0.20)
**Requirements:** 6 locked

## Goal

Paladin desktop distinguishes service health from process ownership so dev mode can attach healthy external Agent/Go services or spawn missing services, while packaged mode keeps a supervisor-owned runtime path that does not depend on dev tools.

## Background

Phase 07.3 added a Rust `ProcessSupervisor` for the Python Agent on port 9876 and the Go Server on port 9880. It loads `apps/desktop/src-tauri/processes.json`, spawns Server then Agent, runs health probes, restarts crashed children, captures logs, exposes Tauri commands, and drives frontend `StartupMask`, `StatusBar`, `ProcessLight`, and the process Zustand store.

The current implementation still treats the desktop app as the only owner. `ProcessState` is limited to `starting | running | degraded | unhealthy | stopped`, `ProcessInfoDTO` has no `owner` or `health` field, and `restart_*` / `stop_*` commands operate as supervisor-owned actions. `spawn_child_to_slot` checks whether a port is free before spawning, but it does not first probe whether the occupied port is a healthy Paladin service that should be attached as external. As a result, a manually started healthy Agent or Go Server can be shown as a desktop process failure or port conflict even though the service is usable.

Phase 07.4 locks the runtime mode semantics before implementation: dev mode is hybrid attach/spawn/conflict, packaged mode is supervisor-owned and dev-tool-free, and UI/actions must reflect ownership rather than child-process assumptions.

## Requirements

1. **Three-dimensional process status**: Agent and Go Server status expose `state`, `owner`, and `health` as separate dimensions, including `state=conflict`, `owner=external|supervisor|none`, and `health=healthy|degraded|failed|unknown`.
   - Current: Rust and frontend status expose only `state`; no `conflict`, `owner`, or explicit `health` exists.
   - Target: `get_process_status` and `process-status` events return all three dimensions for both services; frontend store/types render them without collapsing ownership into health.
   - Acceptance: Automated tests or type-level checks prove the DTO/store accepts `conflict`, `external`, `supervisor`, `none`, `healthy`, `degraded`, `failed`, and `unknown`; repeated identical status updates do not change the status tuple.

2. **Dev hybrid startup resolution**: In `mode=dev`, startup probes configured health endpoints before spawn: healthy services attach as external, absent services on free ports spawn as supervisor-owned, and occupied ports with failed health become conflict without spawn.
   - Current: Supervisor checks port availability first; any occupied port is treated as a spawn blocker.
   - Target: For each service, preflight resolves exactly one path: attach healthy external, spawn absent/free, or conflict occupied/unhealthy.
   - Acceptance: Rust tests cover Agent and Server dev preflight outcomes for healthy external, free-port spawn, and occupied unhealthy conflict; concurrent or repeated redetect/start calls cannot double-spawn one service.

3. **Packaged supervisor-owned mode**: In `mode=packaged`, runtime uses supervisor-owned sidecars from packaged configuration and does not rely on `uv`, `go`, login-shell `PATH`, or repository-relative dev commands; real bundled binary production remains Phase 10.
   - Current: `processes.json` is `mode=dev` and commands are `uv run ...` / `go run ...` with repo-relative cwd.
   - Target: The runtime has a packaged-mode branch/config contract that starts only supervisor-owned configured binaries and rejects or avoids dev command assumptions.
   - Acceptance: Automated tests prove `mode=packaged` does not invoke `uv`, `go`, login-shell path augmentation, or repo-relative dev cwd; packaged status reports `owner=supervisor`; actual frozen binaries and Tauri `externalBin` packaging remain out of scope.

4. **External service action limits**: External services are session-attached only: stop and restart actions are disabled for `owner=external`, redetect remains available, and switching to managed requires the user to stop the external service before re-detection.
   - Current: UI shows restart/stop buttons for each service and commands assume tracked child ownership.
   - Target: `owner=external` disables or rejects stop/restart, labels the service as externally connected, and provides a re-detect path; no preference is persisted.
   - Acceptance: Frontend and command tests prove `stop_*` / `restart_*` do not kill or spawn for `owner=external`; repeated redetect keeps healthy external services attached and reclassifies only when health/port state changes.

5. **Failure classification UI**: `StartupMask`, `StatusBar`, and `ProcessLight` show distinct failure classes for executable missing, cwd missing, port conflict, health failure, startup-grace exit, and readiness degraded, with owner and health labels visible to the user.
   - Current: UI shows starting/running/degraded/unhealthy/stopped, but conflict and ownership are not visible and several startup failures collapse into generic stopped/error text.
   - Target: UI text and actions distinguish external attached, supervisor-owned, conflict, health failure, missing executable, missing cwd, startup-grace exit, and Go readiness degraded.
   - Acceptance: Component/store checks cover all listed failure classes; conflict includes port and health diagnostics; readiness degraded is not rendered as stopped.

6. **Automated verification matrix**: Verification covers Rust state/probe transitions and frontend store/UI behavior for dev attach, dev spawn, conflict, packaged supervisor-owned mode, external action disabling, and failure classification.
   - Current: Phase 07.3 has Rust tests for state machine/config/port/supervisor behavior and TypeScript checks, but no tests for owner/health/conflict/runtime mode semantics.
   - Target: Phase 07.4 adds focused tests for the new runtime-mode contract without requiring full manual UAT as the primary proof.
   - Acceptance: The automated matrix includes at least dev attach, dev spawn, conflict, packaged supervisor-owned, external action disabled, and failure classes; omitting any category fails the phase verification.

## Boundaries

**In scope:**
- `ProcessState` / DTO / frontend store changes needed for `state + owner + health`.
- Dev hybrid preflight: health probe first, attach healthy external, spawn absent/free, conflict occupied/unhealthy.
- Packaged runtime-mode branch/contract that is supervisor-owned and dev-tool-free.
- Redetect behavior for external/session-attached services.
- UI labels/actions for ownership, health, conflict, and failure classes.
- Rust automated tests and frontend type/component checks for the locked runtime matrix.

**Out of scope:**
- Building real frozen Python Agent or Go binaries - Phase 10 Packaging owns binary production.
- Tauri `externalBin`, `.dmg`, `.msi`, or cross-platform installer validation - Phase 10 owns packaging.
- Persisting a user preference for external vs managed startup - dev attach is session-only in this phase.
- A one-click "switch to managed" flow that kills or stops external processes - users stop external services themselves, then redetect.
- Starting PostgreSQL or Redis from Tauri - dependencies remain external; Go `/readyz` reports readiness degradation.
- Remote/multi-machine Agent or Server support - v1 remains local.
- Log rotation/compression and long-running ops hardening - already deferred to Phase 10 ops.

## Constraints

- Dev mode uses HTTP health endpoints as the attach criterion: Agent `/health`; Go `/healthz` for liveness and `/readyz` for readiness.
- A health endpoint returning the configured expected status is sufficient to attach as external; no additional body signature is required in this phase.
- Ports remain fixed: Agent 9876 and Go 9880 unless config changes them; no automatic port search.
- External attachment is session-only and persists nothing.
- `owner=external` services must not be stopped, killed, or restarted by supervisor actions.
- Packaged mode must not depend on `uv`, `go`, login-shell `PATH`, or repository-relative cwd.
- Existing Phase 07.3 restart/backoff/shutdown behavior remains for supervisor-owned children unless explicitly superseded by the owner dimension.
- Response and UI copy should remain Chinese, matching the existing desktop UI.

## Acceptance Criteria

- [ ] `get_process_status` returns Agent and Server status with `state`, `owner`, and `health`.
- [ ] `process-status` events include `state`, `owner`, and `health`, and frontend store/types consume them.
- [ ] `state=conflict` is represented in Rust DTOs and frontend rendering.
- [ ] Repeated identical status updates leave the `(state, owner, health)` tuple stable.
- [ ] In dev mode, a healthy already-running Agent attaches as `owner=external` without spawning.
- [ ] In dev mode, a healthy already-running Go Server attaches as `owner=external` without spawning.
- [ ] In dev mode, an absent service on a free configured port spawns as `owner=supervisor`.
- [ ] In dev mode, an occupied port with failed health becomes `state=conflict`, `owner=none`, and does not spawn.
- [ ] Concurrent or repeated start/redetect for the same service cannot double-spawn.
- [ ] In packaged mode, services report `owner=supervisor`.
- [ ] In packaged mode, runtime does not invoke `uv`, `go`, login-shell path augmentation, or repo-relative dev cwd.
- [ ] `owner=external` disables or rejects `stop_*` and `restart_*` actions.
- [ ] Redetect is available for external services and reclassifies from current health/port state.
- [ ] UI labels distinguish external attached, supervisor-owned, conflict, unhealthy, degraded, and stopped.
- [ ] Startup UI distinguishes executable missing, cwd missing, port conflict, health failure, startup-grace exit, and readiness degraded.
- [ ] Go readiness degraded is not rendered as stopped and does not imply process ownership failure.
- [ ] The automated verification matrix covers dev attach, dev spawn, conflict, packaged supervisor-owned, external action disabled, and failure classes.
- [ ] The verification matrix is non-empty and missing any required category fails phase verification.
- [ ] `cargo test` for desktop process/runtime-mode tests passes.
- [ ] Frontend type/build checks pass for the updated process store and UI states.
- [ ] Phase 07.4 does not implement real frozen binaries, Tauri `externalBin`, installers, or persisted external/managed preference.
