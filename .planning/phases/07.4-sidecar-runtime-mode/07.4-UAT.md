## Tests

### 1. Cold Start And Clean Exit
expected: After old Paladin Agent/Go processes are stopped and the desktop app is started from scratch, the app should enter the main chat UI without requiring "Restart Agent". The Agent status light should become green/running. The Go status light should become running or degraded if PostgreSQL/Redis readiness is unavailable, but it should not fail with a stale port conflict from the previous desktop session. After closing the desktop app, supervisor-owned Agent/Go processes should be stopped so ports 9876 and 9880 are not left occupied by Paladin-spawned processes.
result: pass

### 2. Dev External Attach
expected: If Agent and/or Go are already running and their health endpoints pass before desktop startup, the desktop should attach to them as external services instead of trying to spawn new processes. The status bar should show owner external semantics and should not report stopped/conflict just because the services are not supervisor children.
result: [pending]

### 3. External Action Limits
expected: For an externally attached service, restart and stop actions should be disabled or rejected, while redetect remains available. Clicking disabled controls should not kill or replace the externally managed process.
result: [pending]

### 4. Conflict Classification
expected: If port 9876 or 9880 is occupied by a process that does not pass the configured Paladin health check, redetect/startup should show a conflict diagnostic that includes the affected port and health-check context. It should not collapse into a generic stopped state or attempt to spawn over the occupied port.
result: [pending]

### 5. Go Readiness Degraded
expected: If Go liveness `/healthz` passes but readiness `/readyz` fails because PostgreSQL/Redis is unavailable, the Go service should display degraded health rather than stopped. This degraded state should not block Agent startup or imply that the supervisor failed to own the process.
result: [pending]

### 6. Packaged/Config Diagnostic Boundary
expected: If runtime config is invalid for packaged mode or contains dev-only assumptions such as `uv`, `go`, or repo-relative cwd values, the desktop should surface a readable diagnostic instead of failing before React can render. Real packaged binaries and installer behavior remain deferred to Phase 10.
result: [pending]
