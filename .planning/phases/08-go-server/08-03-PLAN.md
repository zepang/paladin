<objective>
Stand up the data-access tier: install the sqlc CLI, configure it for pgx/v5, author the annotated SQL queries for users/roles/user_roles, generate typed Go query methods, and implement the pgxpool and go-redis connection wrappers with Ping-based health. This plan produces the `*pgxpool.Pool`, `*redis.Client`, and `*sqlcgen.Queries` that the HTTP and auth plans inject.

Purpose: Plan 08-04 needs `pool.Ping` and `redis.Ping` for `/readyz`; plan 08-05 needs the generated `CreateUser`/`GetUserByEmail`/`AssignDefaultRole`/`GetRolesByUserID` methods. Both are blocked until sqlc generates the code and the pool wrapper exists.
Output: `go generate`-clean generated code, a compiling data package, and unit tests proving pool/redis Ping behavior against the compose stack from 08-02.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| DSN/URL -> pool/client | Untrusted connection strings reach pgxpool/redis before queries run. |
| generated SQL -> DB | sqlc-emitted queries are the only SQL that reaches PostgreSQL. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-08-03-01 | Tampering | SQL injection | high | mitigate | sqlc generates parameterized queries exclusively; pgx v5.10.0 fixes placeholder-confusion (RESEARCH Pitfall #3). |
| T-08-03-02 | Information Disclosure | connection failures | low | mitigate | Pool constructors Ping before returning so misconfiguration fails fast at startup, not at first request. |
| T-08-03-SC | Tampering | package installs | low | accept | go-redis v9 and pgx v5.10.0 are checksum-locked via go.sum; sqlc binary installed from verified tag. |
</threat_model>
