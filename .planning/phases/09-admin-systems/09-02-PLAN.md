  <behavior>
    RED cases (all must fail before implementation):
    - Status: a new user has Status Used=0, Limit=cfg.QuotaLimit, Remaining=Limit, ResetAt within window.
    - CheckAndConsume: first call returns Allowed=true, Used=1; subsequent calls increment; Status is consistent with the counter (idempotency edge — Status does not mutate).
    - Over-limit: when counter reaches limit, the next CheckAndConsume returns Allowed=false, Used=limit, Remaining=0; the counter does not increment beyond limit.
    - Concurrency: N=100 concurrent CheckAndConsume with limit=50 yields exactly 50 Allowed=true and 50 Allowed=false; no lost increments, no over-count (the SPEC R5 hard acceptance).
    - Admin bypass: QuotaGate middleware with roles=["admin"] calls c.Next() without touching Redis; the limiter is never invoked.
    - 429 response: a rejected request returns HTTP 429 with error.code="quota_exceeded", X-RateLimit-Remaining=0, and X-RateLimit-Reset headers.
    - Window reset: after window elapses, Status reports Used=0 again (ZREMRANGEBYSCORE evicts old entries).
    - Empty edge: a user with zero usage is allowed on their first request (not rejected).
  </behavior>
  <implementation>Embed script.lua via //go:embed. On NewLimiter, call SCRIPT LOAD to cache the SHA. CheckAndConsume calls EVALSHA with KEYS[1]=quota:{uid}, ARGV=[now_micros, window_start_micros, limit, window_seconds]. Parse the Lua return {allowed_flag, used, limit} into Decision. Compute ResetAt as now + (window - (now_oldest_in_window)). Status does the same ZREMRANGEBYSCORE + ZCARD but skips ZADD (read-only). QuotaGate reads c.Get("userID") and c.Get("roles"); if roles contains "admin", c.Next() and return; else CheckAndConsume, on !Allowed write 429 with headers. Do NOT use INCR (that is fixed-window); do NOT use WATCH/MULTI (retry storms).</implementation>
