# Phase 10.1: macOS/Linux Packaging Workflows - Context

**Gathered:** 2026-07-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 10.1 delivers native GitHub Actions buildability workflows for macOS and Linux that match the existing Windows x64 MSI workflow standard: run the shared release entrypoint, produce installer artifacts, generate a non-secret build manifest, and document that CI artifacts prove buildability only. It does not change runtime provider configuration, installed-app UAT, signing/notarization, or release publishing.

</domain>

<decisions>
## Implementation Decisions

### Workflow Structure
- **D-01:** Keep `.github/workflows/packaging-windows.yml` as an independent, already-verified Windows workflow.
- **D-02:** Add two new independent workflows: `.github/workflows/packaging-macos.yml` and `.github/workflows/packaging-linux.yml`.
- **D-03:** Do not combine macOS/Linux into a matrix workflow and do not merge all three platforms into one packaging workflow. Independent workflow files keep GitHub Actions logs, artifact names, and platform-specific failure diagnosis clear.

### macOS Runner and Target
- **D-04:** Use `macos-14` for the macOS packaging workflow.
- **D-05:** Lock the macOS target to `aarch64-apple-darwin` and set `TARGET_TRIPLE=aarch64-apple-darwin`.
- **D-06:** The macOS workflow expects one `.dmg` installer artifact and one `build-manifest.json`.
- **D-07:** Do not include Intel macOS/x64 coverage in Phase 10.1; this phase aligns with existing arm64 local UAT evidence and `macDmgFileName()` behavior.

### Linux Dependencies and Buildability Boundary
- **D-08:** Put Linux Tauri/bundling apt dependency installation directly in `.github/workflows/packaging-linux.yml`.
- **D-09:** Do not introduce a separate `scripts/setup-linux-packaging-deps.sh` in this phase. There is only one Linux packaging call site, and inline dependencies make GitHub Actions failure logs easier to read.
- **D-10:** Linux CI success requires both installer formats from the same workflow run: exactly one `.AppImage`, exactly one `.deb`, and one `build-manifest.json`.
- **D-11:** Linux remains buildability-only in this phase: no installed-app UAT, distro compatibility matrix, repository publishing, or desktop-environment QA.

### Manifest CLI and Validation Shape
- **D-12:** Change the manifest CLI to accept repeatable `--installer <path>` arguments rather than platform-specific flags.
- **D-13:** Windows passes one MSI installer, macOS passes one DMG installer, and Linux passes two installer paths: AppImage plus deb.
- **D-14:** `build-manifest.mjs` validates required installer counts and extensions based on `target`, failing closed for unsupported targets, missing required Linux installer types, wrong filenames, missing files, and directories.
- **D-15:** Manifest artifact roles should stay installer-format specific enough for audit clarity, e.g. `msi`, `dmg`, `appimage`, and `deb`, while preserving `agent-sidecar` and `server-sidecar` entries.

### Todo Routing
- **D-16:** Do not fold the matched desktop/provider/UI todos into Phase 10.1. They were reviewed but are outside the packaging workflow boundary locked by SPEC.md.

### the agent's Discretion
- Choose exact workflow job names and artifact names, as long as they remain platform-specific and obvious in GitHub Actions.
- Choose the exact Linux apt package list needed for Tauri bundling, keeping it inline and minimal.
- Choose the exact `build-manifest.mjs` internal data structure for target rules, as long as tests cover every SPEC acceptance case.
- Choose whether the manifest CLI preserves backward compatibility for single `--installer` Windows invocation through the same repeatable parser.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `apps/desktop/scripts/release.mjs`: already builds Agent and Server sidecars, stages them using target-triple filenames, audits bundle inputs for `.env*`, invokes Tauri, and has macOS DMG recovery logic.
- `sidecarFileName(logicalName, targetTriple)`: already maps Windows sidecars to `.exe` and non-Windows sidecars without suffix.
- `currentTargetTriple()`: can infer host target when no explicit target is passed, but workflows should set `TARGET_TRIPLE` explicitly for audit clarity.
- `macDmgFileName()`: already maps Node `arm64` to `aarch64` DMG naming.
- `.github/workflows/packaging-windows.yml`: provides the known-good CI shape: checkout, pnpm, Node 22, uv, Python 3.12, Go setup, Rust toolchain, `pnpm install`, `pnpm release -- --target ... --verify`, manifest creation, minimal artifact upload.

### Established Patterns
- Packaging workflows are native-runner proof, not cross-compilation proof.
- `pnpm release` is the single release-oriented entrypoint; platform workflows should not call PyInstaller, Go build, or Tauri independently except through this path.
- Build manifest is non-secret metadata only: commit, runner OS, architecture, target, role, filename, size, SHA-256.
- Release-honesty docs distinguish buildability, installed-app UAT, and release-ready status.
- Existing Node tests use `node:test` with temporary files to prove deterministic manifest output and fail-closed validation.

### Integration Points
- Add `.github/workflows/packaging-macos.yml` with `macos-14`, `TARGET_TRIPLE=aarch64-apple-darwin`, and one-DMG artifact collection.
- Add `.github/workflows/packaging-linux.yml` with inline apt dependency installation, `TARGET_TRIPLE=x86_64-unknown-linux-gnu` unless implementation research finds a project-specific reason to choose another native Linux target, and AppImage+deb artifact collection.
- Extend `build-manifest.mjs` argument parsing to retain multiple `--installer` values.
- Extend `build-manifest.mjs` target rules to validate sidecar names and installer extensions per platform.
- Update docs to include macOS/Linux buildability workflow status without claiming signing, notarization, installed-app UAT, or release-ready approval.

</code_context>

<specifics>
## Specific Ideas

- Keep the Windows workflow stable and avoid touching it unless required for repeatable `--installer` compatibility or shared manifest behavior.
- Prefer platform-specific workflow names and artifact names so the GitHub Actions UI is easy to scan.
- Treat Linux AppImage and deb as a pair: a run that produces only one of them is a failure, not a partial success.
- Repeatable `--installer` is intentionally generic: workflow scripts stay responsible for locating the expected files, while the manifest generator validates target-specific requirements.

</specifics>
