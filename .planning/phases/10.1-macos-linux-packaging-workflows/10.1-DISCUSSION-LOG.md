## Workflow Split

| Option | Description | Selected |
|--------|-------------|----------|
| Two new workflows | Add `.github/workflows/packaging-macos.yml` and `.github/workflows/packaging-linux.yml`; preserve Windows workflow; logs and artifacts stay platform-specific. | ✓ |
| One matrix workflow | Add one Unix workflow using a macOS/Linux matrix; fewer files but mixed platform dependencies and artifact logic. | |
| Three-platform unified workflow | Merge Windows/macOS/Linux into one packaging workflow; superficially unified but higher risk to existing Windows proof. | |
| Other | User specifies naming or structure. | |

**User's choice:** Two new workflows.
**Notes:** User selected option 1. The key rationale is clarity and low risk: macOS/Linux differ enough that separate workflow files make failures easier to diagnose and avoid disturbing the verified Windows workflow.

---

## macOS Runner and Architecture

| Option | Description | Selected |
|--------|-------------|----------|
| `macos-14` / arm64 | Lock to Apple Silicon runner and `aarch64-apple-darwin`, aligning with existing macOS UAT and DMG naming. | ✓ |
| `macos-latest` | Less configuration, but runner architecture/image may drift over time. | |
| arm64 + x64 macOS | Broader coverage, but higher CI cost and complexity beyond the minimal buildability target. | |
| Other | User specifies runner or architecture strategy. | |

**User's choice:** `macos-14` / arm64.
**Notes:** User selected option 1. Intel macOS coverage remains deferred.

---

## Linux Dependencies

| Option | Description | Selected |
|--------|-------------|----------|
| Inline apt dependencies | Put Linux Tauri/bundling dependencies directly in `packaging-linux.yml`; failures are visible in Actions logs. | ✓ |
| Setup script | Add `scripts/setup-linux-packaging-deps.sh`; reusable but creates another file to inspect. | |
| Hybrid | Keep core dependencies in workflow and complex probing in a script. | |
| Other | User specifies dependency management approach. | |

**User's choice:** Inline apt dependencies.
**Notes:** User selected option 1. This phase has only one Linux packaging call site, so inline dependencies keep the workflow direct.

---

## Manifest CLI Shape

| Option | Description | Selected |
|--------|-------------|----------|
| Repeatable `--installer` | Use `--installer <path>` one or more times; target rules validate required formats/counts. | ✓ |
| Platform-specific flags | Use flags like `--msi`, `--dmg`, `--appimage`, `--deb`; explicit but tightly couples CLI to platforms. | |
| JSON input | Pass a manifest input JSON file; expressive but too heavy for current needs. | |
| Other | User specifies CLI shape. | |

**User's choice:** Repeatable `--installer`.
**Notes:** User selected option 1. Windows/macOS pass one installer; Linux passes AppImage and deb. `build-manifest.mjs` should validate platform-specific requirements from the target.
