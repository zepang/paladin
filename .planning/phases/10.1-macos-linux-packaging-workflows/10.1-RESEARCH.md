## Implementation Decisions

### Workflow Structure
- **D-01:** Keep `.github/workflows/packaging-windows.yml` as an independent, already-verified Windows workflow.
- **D-02:** Add two new independent workflows: `.github/workflows/packaging-macos.yml` and `.github/workflows/packaging-linux.yml`.
- **D-03:** Do not combine macOS/Linux into a matrix workflow and do not merge all three platforms into one packaging workflow. Independent workflow files keep GitHub Actions logs, artifact names, and platform-specific failure diagnosis clear.

### macOS Runner and Target
- **D-04:** Use `macos-14` for the macOS packaging workflow.
- **D-05:** Lock the macOS target to `aarch64-apple-darwin` and set `TARGET_TRIPLE=aarch64-apple-darwin`.
- **D-06:** The macOS workflow expects one `.dmg` installer artifact and one `build-manifest.json`.
- **D-07:** Do not include Intel macOS/x64 coverage in Phase 10.1; this phase aligns with existing arm64 local UAT evidence and `macDmgFileName()` behavior.

### Linux Dependencies and Buildability Boundary
- **D-08:** Put Linux Tauri/bundling apt dependency installation directly in `.github/workflows/packaging-linux.yml`.
- **D-09:** Do not introduce a separate `scripts/setup-linux-packaging-deps.sh` in this phase. There is only one Linux packaging call site, and inline dependencies make GitHub Actions failure logs easier to read.
- **D-10:** Linux CI success requires both installer formats from the same workflow run: exactly one `.AppImage`, exactly one `.deb`, and one `build-manifest.json`.
- **D-11:** Linux remains buildability-only in this phase: no installed-app UAT, distro compatibility matrix, repository publishing, or desktop-environment QA.

### Manifest CLI and Validation Shape
- **D-12:** Change the manifest CLI to accept repeatable `--installer <path>` arguments rather than platform-specific flags.
- **D-13:** Windows passes one MSI installer, macOS passes one DMG installer, and Linux passes two installer paths: AppImage plus deb.
- **D-14:** `build-manifest.mjs` validates required installer counts and extensions based on `target`, failing closed for unsupported targets, missing required Linux installer types, wrong filenames, missing files, and directories.
- **D-15:** Manifest artifact roles should stay installer-format specific enough for audit clarity, e.g. `msi`, `dmg`, `appimage`, and `deb`, while preserving `agent-sidecar` and `server-sidecar` entries.

### Todo Routing
- **D-16:** Do not fold the matched desktop/provider/UI todos into Phase 10.1. They were reviewed but are outside the packaging workflow boundary locked by SPEC.md.

### the agent's Discretion
- Choose exact workflow job names and artifact names, as long as they remain platform-specific and obvious in GitHub Actions.
- Choose the exact Linux apt package list needed for Tauri bundling, keeping it inline and minimal.
- Choose the exact `build-manifest.mjs` internal data structure for target rules, as long as tests cover every SPEC acceptance case.
- Choose whether the manifest CLI preserves backward compatibility for single `--installer` Windows invocation through the same repeatable parser.

## Summary

Phase 10.1 should be planned as a narrow CI/build-manifest extension, not as a new packaging architecture. The correct implementation path is to preserve the Windows workflow contract, add two independent native workflows, keep `pnpm release -- --target ... --verify` as the only build entrypoint, and stage only platform installer files plus `build-manifest.json` for upload. [CITED: .planning/phases/10.1-macos-linux-packaging-workflows/10.1-CONTEXT.md] [VERIFIED: codebase]

The main implementation work is in `apps/desktop/scripts/build-manifest.mjs`: replace the Windows-only target constant with explicit target rules for `x86_64-pc-windows-msvc`, `aarch64-apple-darwin`, and `x86_64-unknown-linux-gnu`; parse repeatable `--installer`; validate sidecar filenames and installer roles per target; and keep manifest output deterministic and non-secret. [CITED: .planning/phases/10.1-macos-linux-packaging-workflows/10.1-SPEC.md] [VERIFIED: codebase]

**Primary recommendation:** Use explicit native workflows with platform-local installer staging directories: `macos-artifact/*.dmg + build-manifest.json` and `linux-artifact/*.AppImage + *.deb + build-manifest.json`; fail before upload unless exact counts are met. [CITED: .planning/phases/10.1-macos-linux-packaging-workflows/10.1-SPEC.md] [VERIFIED: codebase]

## Architecture Patterns

### System Architecture Diagram

```text
workflow_dispatch / packaging PR path
  -> platform workflow
     -> setup Node + pnpm + uv + Python + Go + Rust
     -> Linux only: apt install Tauri/WebKitGTK bundling dependencies
     -> set TARGET_TRIPLE
     -> pnpm install --frozen-lockfile
     -> pnpm release -- --target $TARGET_TRIPLE --verify
        -> PyInstaller Agent sidecar
        -> Go Server sidecar
        -> .env bundle input audit
        -> stage sidecars as paladin-*-sidecar-$TARGET_TRIPLE[.exe]
        -> pnpm tauri build
        -> macOS only: recover headless DMG if Tauri bundler leaves app bundle + bundle_dmg.sh
     -> locate exact installer outputs
        -> if wrong count or missing format: fail
        -> else copy installers into platform-artifact/
     -> node build-manifest.mjs --installer ... --installer ...
        -> validate target rules
        -> hash sidecars and installers
        -> write deterministic non-secret manifest
     -> upload platform-artifact installers + build-manifest.json only
```

### Pattern 1: Target Rule Table

**What:** Encode supported targets as data, not nested special cases. [VERIFIED: codebase]

**When to use:** Use in `build-manifest.mjs` for sidecar filename regex, architecture, runner expectations, installer role requirements, and count validation. [CITED: .planning/phases/10.1-macos-linux-packaging-workflows/10.1-SPEC.md]

**Example:**

```js
// Source: codebase pattern + Phase 10.1 SPEC
const TARGET_RULES = {
  'x86_64-pc-windows-msvc': {
    architecture: 'x86_64',
    sidecarSuffix: 'x86_64-pc-windows-msvc.exe',
    installers: [{ role: 'msi', pattern: /\.msi$/i, required: 1 }],
  },
  'aarch64-apple-darwin': {
    architecture: 'aarch64',
    sidecarSuffix: 'aarch64-apple-darwin',
    installers: [{ role: 'dmg', pattern: /\.dmg$/i, required: 1 }],
  },
  'x86_64-unknown-linux-gnu': {
    architecture: 'x86_64',
    sidecarSuffix: 'x86_64-unknown-linux-gnu',
    installers: [
      { role: 'appimage', pattern: /\.AppImage$/ },
      { role: 'deb', pattern: /\.deb$/i },
    ],
  },
};
```

### Pattern 2: Repeatable CLI Parser

**What:** Preserve all `--installer <path>` occurrences in order and reject missing values. [CITED: .planning/phases/10.1-macos-linux-packaging-workflows/10.1-CONTEXT.md]

**When to use:** Needed for Linux AppImage+deb while preserving Windows single-installer invocation. [CITED: .planning/phases/10.1-macos-linux-packaging-workflows/10.1-SPEC.md]

**Example:**

```js
// Source: Phase 10.1 SPEC
function parseArgs(argv) {
  const options = { installers: [] };
  for (let index = 0; index < argv.length; index += 1) {
    const key = argv[index];
    const value = argv[index + 1];
    if (!key?.startsWith('--') || value === undefined) throw new Error(`Invalid argument: ${key ?? ''}`);
    index += 1;
    if (key === '--installer') options.installers.push(value);
    else options[key.slice(2)] = value;
  }
  return options;
}
```

### Anti-Patterns to Avoid

- **Bypassing `pnpm release`:** Direct PyInstaller/Go/Tauri calls inside workflows would split the release path and bypass existing `.env` and sidecar staging audits. [VERIFIED: codebase] [CITED: .planning/phases/10.1-macos-linux-packaging-workflows/10.1-SPEC.md]
- **Uploading `src-tauri/target/**`:** Full build trees can include logs, caches, dependency files, and unreviewed outputs; stage exact installers into a clean artifact directory first. [CITED: .planning/phases/10.1-macos-linux-packaging-workflows/10.1-SPEC.md]
- **Inferring release-ready from CI success:** CI artifact success proves buildability only for macOS/Linux in this phase. [CITED: .planning/phases/10.1-macos-linux-packaging-workflows/10.1-SPEC.md]
- **Using `ubuntu-latest` for Linux evidence:** `-latest` can drift, and Tauri AppImage compatibility depends on base system/glibc. Prefer explicit `ubuntu-22.04` unless planner documents a reason otherwise. [CITED: https://docs.github.com/en/actions/reference/runners/github-hosted-runners] [CITED: https://v2.tauri.app/distribute/appimage/]

## Common Pitfalls

### Pitfall 1: Linux workflow succeeds with only one installer
**What goes wrong:** AppImage or deb is missing but the workflow still uploads a partial artifact. [CITED: .planning/phases/10.1-macos-linux-packaging-workflows/10.1-SPEC.md]
**Why it happens:** Globs are permissive unless explicit count checks run before upload. [ASSUMED]
**How to avoid:** Add shell/Node count checks before manifest generation and make manifest target rules require both roles. [CITED: .planning/phases/10.1-macos-linux-packaging-workflows/10.1-SPEC.md]
**Warning signs:** Upload path uses broad `bundle/**` or lacks `if-no-files-found: error`. [VERIFIED: codebase]

### Pitfall 2: macOS DMG path differs from local arm64 expectation
**What goes wrong:** The workflow cannot find the DMG or finds multiple files after recovery. [VERIFIED: codebase]
**Why it happens:** `release.mjs` can recover a headless DMG via `bundle_dmg.sh`, and stale/rw files may exist if not cleaned. [VERIFIED: codebase]
**How to avoid:** Use exact `target/release/bundle/dmg/*.dmg` count check after `pnpm release`, then stage exactly one DMG. [CITED: .planning/phases/10.1-macos-linux-packaging-workflows/10.1-SPEC.md]
**Warning signs:** Workflow copies from `bundle/macos` or uploads both `.app` and `.dmg`. [ASSUMED]

### Pitfall 3: Linux apt list is too small
**What goes wrong:** Tauri build fails on Ubuntu before producing AppImage/deb. [CITED: https://v2.tauri.app/start/prerequisites/]
**Why it happens:** WebKitGTK 4.1 and desktop integration libraries are system packages, not npm packages. [CITED: https://v2.tauri.app/start/prerequisites/]
**How to avoid:** Inline apt install at least `libwebkit2gtk-4.1-dev`, `build-essential`, `curl`, `wget`, `file`, `libxdo-dev`, `libssl-dev`, `libayatana-appindicator3-dev`, `librsvg2-dev`, plus official workflow additions `libappindicator3-dev`, `patchelf`, and `xdg-utils` if needed by the runner image. [CITED: https://v2.tauri.app/start/prerequisites/] [CITED: https://v2.tauri.app/distribute/pipelines/github/]
**Warning signs:** `pkg-config` or WebKitGTK errors in CI. [ASSUMED]

### Pitfall 4: Manifest leaks environment or logs
**What goes wrong:** Manifest includes secret env values, DSNs, tokens, or copied logs. [CITED: .planning/phases/10.1-macos-linux-packaging-workflows/10.1-SPEC.md]
**Why it happens:** Reading from `process.env` or serializing workflow context wholesale is easier than passing explicit fields. [ASSUMED]
**How to avoid:** Only accept `commit`, `runner-os`, `target`, `agent`, `server`, repeatable `installer`, and `output`; write fixed schema fields plus filename/size/SHA-256. [VERIFIED: codebase]
**Warning signs:** Tests do not set a fake secret and assert absence from `JSON.stringify(manifest)`. [VERIFIED: codebase]
