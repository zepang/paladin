# Phase 10.1: macOS/Linux Packaging Workflows — Specification

**Created:** 2026-07-13
**Ambiguity score:** 0.19 (gate: ≤ 0.20)
**Requirements:** 5 locked

## Goal

Extend the existing Windows native packaging CI standard to macOS and Linux so pull-request or manual CI can prove buildability by producing audited installer artifacts and non-secret manifests for each platform.

## Background

Phase 10 completed the desktop packaging baseline and proved Windows x86_64 MSI buildability through `.github/workflows/packaging-windows.yml`. That workflow runs a native `windows-latest` build, invokes the shared `pnpm release -- --target x86_64-pc-windows-msvc --verify` entrypoint, uploads exactly one MSI plus `build-manifest.json`, and documents that the artifact proves buildability only.

The codebase already has reusable release machinery in `apps/desktop/scripts/release.mjs`: it builds the Python Agent sidecar with PyInstaller, builds the Go Server sidecar, stages Tauri external binaries by target triple, invokes `pnpm tauri build`, and includes macOS DMG recovery for headless CI. The manifest generator is still Windows-only: `apps/desktop/scripts/build-manifest.mjs` accepts only `x86_64-pc-windows-msvc`, requires exactly one MSI installer path, and validates Windows `.exe` sidecar names. Linux CI packaging does not exist yet, and docs currently distinguish buildability from installed-app UAT/release-ready status.

## Requirements

1. **macOS native packaging workflow**: CI must build macOS installer artifacts through the same release entrypoint used by Windows.
   - Current: macOS can be built locally through `pnpm release`, and `release.mjs` has DMG recovery logic, but there is no dedicated GitHub Actions workflow that uploads a macOS artifact plus manifest.
   - Target: A macOS workflow runs on a native macOS runner, builds the desktop app through `pnpm release -- --verify`, locates the generated DMG, creates a non-secret manifest, and uploads only the DMG plus manifest.
   - Acceptance: A manually triggered macOS workflow run succeeds on a supported macOS runner and its artifact contains exactly one `.dmg` and one `build-manifest.json`.

2. **Linux native packaging workflow**: CI must build Linux AppImage and deb artifacts through the same release entrypoint used by Windows.
   - Current: Linux is not represented in packaging CI, and no workflow proves that Linux Tauri bundles can be produced with packaged Agent and Go sidecars.
   - Target: A Linux workflow runs on a native Linux runner, installs required system dependencies for Tauri bundling, invokes `pnpm release -- --verify`, locates both AppImage and deb outputs, creates a non-secret manifest, and uploads only those installers plus manifest.
   - Acceptance: A manually triggered Linux workflow run succeeds and its artifact contains exactly one `.AppImage`, exactly one `.deb`, and one `build-manifest.json`.

3. **Cross-platform build manifest**: `build-manifest.mjs` must validate Windows, macOS, and Linux targets and support one or more installer artifacts per platform.
   - Current: The manifest generator rejects every target except `x86_64-pc-windows-msvc`, expects one `installer` path, hard-codes Windows sidecar filenames, and labels the installer role as `msi`.
   - Target: The manifest generator accepts the supported target triples for Windows x86_64, macOS native CI, and Linux x86_64; validates sidecar filename suffixes per platform; validates installer extensions per platform; supports multiple installer paths where Linux requires AppImage plus deb; and emits deterministic, non-secret metadata.
   - Acceptance: Node tests cover Windows MSI, macOS DMG, Linux AppImage+deb, missing files, directories, unsupported target, wrong sidecar filename, wrong installer filename, and missing required Linux installer type.

4. **Artifact boundary remains minimal and auditable**: Packaging workflows must upload installer artifacts and manifest only, never the whole build tree or sidecar binaries as standalone artifacts.
   - Current: The Windows workflow already copies only the MSI into `windows-artifact` and uploads that file plus `build-manifest.json`.
   - Target: macOS and Linux workflows use the same minimal artifact boundary: installer files plus `build-manifest.json`; sidecar binaries are represented in the manifest by filename, size, and SHA-256 but are not uploaded separately outside the installers.
   - Acceptance: Workflow artifact upload paths for Windows, macOS, and Linux include only installer file globs and `build-manifest.json`, and exclude `src-tauri/target/**`, `src-tauri/binaries/**`, `.env*`, logs, caches, and dependency directories.

5. **Documentation states platform status honestly**: Project docs must describe macOS and Linux packaging workflows as buildability checks, not release approval.
   - Current: `docs/packaging.md` documents Windows buildability and existing platform status, while Linux remains outside the implemented packaging workflow story.
   - Target: `README.md` and/or `docs/packaging.md` describe Windows, macOS, and Linux CI artifacts, their installer formats, manifest contents, and the distinction between buildability, installed-app UAT, and release-ready status.
   - Acceptance: Documentation explicitly says successful CI artifacts for macOS/Linux do not imply signed/notarized packages, installed-app UAT completion, or release-ready approval.

## Boundaries

**In scope:**
- Add native GitHub Actions workflow coverage for macOS packaging buildability.
- Add native GitHub Actions workflow coverage for Linux packaging buildability.
- Extend `apps/desktop/scripts/build-manifest.mjs` and its tests for Windows/macOS/Linux and multi-installer Linux output.
- Preserve the existing Windows MSI workflow behavior and artifact boundary.
- Update packaging documentation so platform CI status and release-ready status are not conflated.

**Out of scope:**
- macOS code signing, notarization, staple verification, Gatekeeper distribution readiness — this phase proves unsigned buildability only.
- Windows installed-app UAT or Windows release-ready approval — Phase 10 intentionally left that deferred.
- Linux installed-app UAT, distro compatibility matrix, package repository publishing, sandbox integration, or desktop-environment QA — this phase only proves CI artifact generation.
- Automatic GitHub Releases, release notes, version bumping, publishing, or long-term retention policy changes — artifact upload remains CI-scoped.
- Provider configuration, runtime environment UX, database/Redis dependency behavior, or packaged app startup semantics — those belong to separate runtime configuration work.

## Constraints

- Workflows must use native runners for their target platform; cross-compilation is not accepted as proof for this phase.
- The release entrypoint remains `pnpm release`; workflows should not invent a second packaging path that bypasses sidecar build/staging audits.
- Manifest content must remain non-secret: no environment variable values, API keys, DSNs, Redis passwords, JWT secrets, tokens, or copied logs.
- Linux packaging must produce both AppImage and deb in the same successful buildability workflow.
- Workflow path triggers should stay scoped to packaging-relevant files to avoid broad CI noise.

## Acceptance Criteria

- [ ] Existing Windows workflow still produces one MSI plus `build-manifest.json` and keeps its minimal upload boundary.
- [ ] macOS workflow can be manually triggered and produces exactly one DMG plus `build-manifest.json`.
- [ ] Linux workflow can be manually triggered and produces exactly one AppImage, exactly one deb, and `build-manifest.json`.
- [ ] `apps/desktop/scripts/build-manifest.test.mjs` covers Windows, macOS, Linux multi-installer output, unsupported targets, bad filenames, missing files, directories, and missing required Linux installer types.
- [ ] Manifest JSON for every supported platform records `schema_version`, `commit_sha`, `runner_os`, `architecture`, `target`, and artifact entries with role, filename, size, and SHA-256.
- [ ] Workflow upload paths do not include full build directories, dependency caches, standalone sidecar binaries, `.env*`, logs, or secrets-bearing files.
- [ ] Packaging docs distinguish CI buildability from installed-app UAT and release-ready status for macOS and Linux.
- [ ] Successful macOS/Linux packaging CI is not described anywhere as signed, notarized, installed-app verified, or release-ready.
